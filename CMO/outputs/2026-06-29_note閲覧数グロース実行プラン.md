- **作成日**: 2026-06-29
- **担当**: CMO（code）
- **対象アカウント**: https://note.com/safe_canna441 （公開334本・2026-06-23 棚卸し時点）
- **依頼**: オーナー「noteの閲覧数伸ばして」

  数字（閲覧数・スキ）はオーナー画面でしか見えない。**code＝中身の最適化／owner・cowork＝投稿と編集**。

## 要点（結論）

note の閲覧数は **①検索で見つかる ②回遊で連れていく ③フォロワーに毎回届く** の3経路でしか増えない。
334本という資産はもう十分にある。**伸びていない最大の理由は「新規作成」ではなく、既存記事の
「見つかりやすさ（タイトル・タグ）」と「束ね方（マガジン・回遊）」が最適化されていないこと。**

→ ROI が高い順に5レバー。**L1・L2 は code が今すぐ放置で回せる。L3 は owner 10分の一度きり作業で構造的に効く。**

| # | レバー | 効く経路 | 担当 | owner作業量 | 即効性 |
|---|--------|---------|------|-----------|--------|
| **L1** | タイトル・タグに検索キーワードを補う | 検索 | code が改善案→owner編集 | 中（一括編集） | 中（数週で積む） |
| **L2** | 回遊フッター（あわせて読む＋フォロー導線）を全記事に | 回遊・フォロー | code（既存システム）→cowork貼付 | 小 | 中 |
| **L3** | マガジン編成（334本をテーマ別に束ねる） | 回遊・検索 | owner 一度きり | 小（10〜20分） | 高 |
| **L4** | 投稿時間・頻度・タグの定番化 | フォロー・検索 | owner/cowork 定常 | 小 | 低〜中 |
| **L5** | 高ポテンシャル記事のタイトルだけリライト | 検索・回遊 | owner 編集 | 小 | 中 |

> 正直な弱点：閲覧数の伸びは数週〜数ヶ月で積むタイプで、即日では跳ねない。
> 「今日やって明日2倍」になる魔法は無い。ただし **L1〜L3 はやるほど複利で効き、放置で残る。**

## L1：タイトル・タグに「検索キーワード」を補う（最重要・放置で効く）

### なぜ効くか
note 記事は **note 内検索と Google 検索の両方に乗る**。ローカル記事（高岡・氷見・富山の食/場所）は
競合が薄く、`高岡 朝市`・`氷見 うどん`・`五箇山 豆腐` のような**地名＋対象**の検索で上位を取りやすい。
ところが現状のタイトルは叙情的で、**検索されるキーワードがタイトルに無い／弱い**ものが多い。

| 現状タイトル | 検索される語 | 問題 |
|---|---|---|
| 土曜の朝、家のすぐ近くの市場まで歩く。 | 高岡 朝市 / 富山 朝市 | 「市場」「高岡」がタイトルに無く検索流入ゼロ |
| ころ柿は、軒先で甘くなる。 | 富山 ころ柿 / 干し柿 | 地名が無い |
| たら汁は、漁師の体をあたためる鍋。 | 朝日町 たら汁 / 富山 たら汁 | 地名が無い |
| 金屋町は、鋳物が始まった町。 | 高岡 金屋町 | 「高岡」が無い |

### 改善ルール（今後の標準書式に組込み）
1. **タイトルに「地名（市レベル）＋固有名詞」を必ず1つ入れる**（A4順守＝町名・住所はNG、市まで）。
   叙情フックは残してOK。例：`土曜の朝、家のすぐ近くの市場まで歩く。` →
   `高岡の朝市まで、土曜の朝は歩いて買いに行く。`（フック＋「高岡」「朝市」を前方に）
2. **ダッシュ後置パターンは継続**（既に良い）：`— 富山「大門素麺」` のように固有名詞を併記。
3. **冒頭2〜3行で検索意図に即答**（note は冒頭がプレビュー＝検索結果のスニペットになる）。
   「これは何の記事か」を1行目で名指しする（地名＋対象）。
4. **ハッシュタグは "検索される定番タグ" を最低3つ固定**：`#富山` `#高岡` `#富山旅行`／食なら `#富山グルメ`
   `#ご当地グルメ`／場所なら `#富山観光` `#北陸旅行`。叙情タグ（#土曜の朝 等）は4つ目以降に。

→ **具体的な改善バッチ＝別ファイル `2026-06-29_note_タイトル最適化バッチ.md`**（owner が note 編集画面で
タイトルを差し替えるだけのリスト）。

## L2：回遊フッターを全記事に行き渡らせる（複利・既存システム活用）

- 既に `CDO/outputs/note_footer/gen_footers.py` が「▼あわせて読む（関連2本）＋フォロー導線」を生成する。
  これが付いた記事は **1人の読者が2〜3本読む（PV増）＋フォローに変わる（将来のPV増）**。
- 現状 `published_registry.tsv` は **14件しか登録が無い**（公開334本に対し穴だらけ）。
  → registry を充実させるほど関連リンクの精度が上がる。**ただし note_id の取得は note API が要る＝A1で code 不可。**
  cowork が棚卸し時の API 取得（`/api/v2/creators/safe_canna441/contents`）で全 note_id を registry に流し込めば、
  全記事に質の高い回遊フッターを後付けできる。
- **owner/cowork アクション**：①registry を334本分に更新（cowork・API1回）②未フッター記事に footer を貼る（cowork バッチ）。

## L3：マガジン編成（owner 10〜20分・構造的に効く一手）

note の「マガジン」は **クリエイターページの回遊を激増させる**（テーマで束ねると関連記事が芋づる式に読まれる）。
334本がバラ積みの今、これが最大の構造的伸びしろ。**code が編成案を出し、owner が note UI で作るだけ。**

### 推奨マガジン構成（既存記事の2系統に沿う）

| マガジン名（案） | 中身 | 狙う読者 |
|---|---|---|
| **富山・高岡の食、住人の食卓から** | 食記事（白えび・ブリ・昆布じめ・コロッケ・大門素麺 等） | ご当地グルメ・旅行検索 |
| **歩いて行ける富山・高岡** | 場所記事（金屋町・雨晴・内川・庄川峡・朝市 等） | 観光・散歩・移住検討 |
| **氷見・高岡 まんが聖地さんぽ** | 藤子・笑ゥせぇるすまん・潮風ギャラリー 等 | アニメ聖地巡礼（海外含む） |
| **ひとり会社の続け方** | フリーランス/AI/仕事術 系 | 副業・独立層（別ペルソナ） |
| **Toyama for English readers** | 英語要約が強い記事／英語版 | 海外読者（北極星） |

> 食と場所と暮らし／仕事は**読者層が別**。マガジンで分けると、各マガジンが
> 独立した入口になり、検索・回遊の面が増える（＝表示機会が増える）。

## L4：投稿時間・頻度・タグの定番化（定常運用・owner/cowork）

- **投稿時間**：生活×ローカルの読者は **平日朝7〜8時台／夜21〜23時台** に動く。公開時刻をここに寄せる。
- **頻度の一貫性**：note のおすすめは「継続的に出すアカウント」を優遇する傾向。**毎日1本でなくていいので
  "途切れさせない"**（週数本でも定期）。既に日次に近い制作量があるので、公開ペースを均すだけ。
- **スキ周り（相互回遊）**：同テーマ（#富山 #フリーランス）の新着に少しスキを付けると、相手とその読者に
  アカウントが露出する。owner/cowork が1日5分でも効く（規約内・自然な範囲で）。

---

## L5：高ポテンシャル記事のタイトルだけリライト（owner 編集）

既に公開済みで**内容は良いが検索で埋もれている記事**は、タイトル差し替えだけで掘り起こせる。
note は公開後もタイトル編集が可能（URLは不変＝既存の評価を失わない）。
→ L1 の改善バッチのうち、**反応が見たい上位10本だけ先に owner が編集**して効果を見るのが安全。

## 担当の切り分け（正直版）

| アクション | 担当 | 理由 |
|---|---|---|
| タイトル・タグ・冒頭の改善案作成、書式更新、回遊ロジック | **code** | A1非該当（中身の制作） |
| 既存記事のタイトル/タグの実編集、マガジン作成、公開、スキ周り | **owner / cowork** | A1（note UI操作）＝code不可 |
| 閲覧数・スキ・流入元の数値確認 | **owner** | A1（管理画面）＝code不可 |
| registry を334本分に更新（note API） | **cowork** | A1（外部API）＝code不可 |

> **数値が見えないと検証できない**のが構造的弱点。最低限 **L5 の上位10本だけでも owner が編集→1〜2週後に
> 閲覧数を code に共有**してくれれば、効いた型に寄せて残りを最適化できる（小さく検証→拡大）。
