# 設計：cowork 主導の指示フロー（2026-06-11）【破棄】

## 目的
これまで「code が指示を出し cowork が実行」だったが、cowork が文脈を持てず混乱した。
**cowork を指示する側（オーケストレーター）に、code を実装・報告する側に反転**する。
理由：cowork は **ネット可・note/X/画像サイトに到達でき、オーナーと同じ環境で動く**＝
全体状況を把握して「何をやるか」を決めるのに向く。code は **クラウド・ネット遮断**で、
記事制作・ツール作成・自動化の「手を動かす」実装に専念するのが自然。

---

## 役割（反転後）

### cowork ＝ オーケストレーター（指示する側）
- オーナーの意図（自然言語）を受け取り、タスクに分解する。
- **実行系を担う**：note公開・サムネ挿入・X等クロスポスト・フリー画像の手配・ブラウザ操作。
- code への作業指示を `ops/inbox/<日付>_<連番>_cowork_code.yaml` に書く。
- code の納品（outbox）を受けて、公開などの最終アクションを実行し、オーナーへ報告。

### code ＝ 実装・報告する側
- `ops/inbox/*_cowork_code.yaml` の指示を読み、実装する：
  記事制作・台帳/STATE整備・スクリプト/ワークフロー作成・多言語化など。
- ネット遮断のため**外部送信（note/X/画像取得）はしない**。クラウドで完結する作業に限定。
- 成果は commit & push し、`ops/outbox/<日付>_<連番>_code_cowork.yaml` で完了報告。

---

## フロー
```
オーナー
  │ 自然言語で依頼
  ▼
cowork（指示する側）── ops/inbox/*_cowork_code.yaml ──▶ code（実装する側）
  ▲                                                        │ 実装 → commit/push
  │ ops/outbox/*_code_cowork.yaml（完了報告）◀─────────────┘
  │
  └─ 公開/サムネ/X など実行系を cowork が担当 → オーナーへ報告
```

## 指示YAMLの形式（cowork → code）
`ops/inbox/2026-06-11_001_cowork_code.yaml`
```yaml
---
id: 2026-06-11_001
from: cowork
to: code
created: <ISO8601>
priority: normal|high|urgent
type: instruction
status: open
title: <一行で何をしてほしいか>
---
## 依頼
- 具体的に：対象・本数・締切・制約（無料のみ等）
## 完了条件
- 何があれば完了か（ファイル・台帳反映・push）
## 補足
- 背景・参照ファイル
```

## 報告YAMLの形式（code → cowork）
`ops/outbox/2026-06-11_001_code_cowork.yaml`
```yaml
---
id: 2026-06-11_001
from: code
to: cowork
re: 2026-06-11_001
status: done|blocked
title: <結果一行>
---
## やったこと
- 作成/変更ファイル一覧（パス）
- commit ハッシュ / push 先ブランチ
## cowork側の残作業
- 公開・サムネ挿入・X投稿など（code には不可能な実行系）
## 詰まり（あれば）
- 何が不足か・判断が必要な点
```

## 運用ルール
- **指示の起点は cowork**（オーナー意図の翻訳・分解は cowork が行う）。
- code は指示が無くても「制作の作り置き」は自律で進めてよい（従来どおり）。ただし
  公開に関わる判断・実行は cowork の指示/担当とする。
- 1指示=1報告を基本に、ops/ で履歴を残す（削除しない）。
- 機微情報（APIキー等）は YAML に書かない。キーは GitHub Secrets / ローカルのみ。
