# 設計：元請=code / 下請=cowork（2026-06-11 オーナー指示）

## 体制（請負構造）
```
オーナー（発注者）
   │ 依頼（自然言語）
   ▼
code（元請・PRIME）  ── 作業指示 ──▶  cowork（下請・SUB）
   ▲                                      │ ネット必須の実行
   └────── 完了報告 ◀─────────────────────┘
   │
   └─ 統括して オーナーへ報告（責任を持つ）
```

- **オーナー＝発注者**。依頼の窓口は **code（元請）一本**。
- **code＝元請（PRIME）**：受注・全体設計・タスク分解・指示・統括・最終責任。
- **cowork＝下請（SUB）**：code の指示を受け、**ネットが要る実行**を代行。

---

## code（元請）の責務
1. **受注**：オーナーの意図を受け取る唯一の窓口。
2. **設計・分解**：何を作り・どう出すかを設計し、タスクに分ける。
3. **自前実装（クラウドで完結する全て）**：
   記事制作・台帳/STATE整備・スクリプト/ワークフロー・多言語化・X投稿文生成・サムネ自動取得WF。
4. **下請への指示発行**：`ops/inbox/<日付>_<連番>_code_cowork.yaml` に work order を書く。
5. **統括・報告**：cowork の報告を取り込み、台帳へ反映し、**オーナーへ一本化して報告**。
   うまくいかない時の原因究明・再設計も元請の責任。
6. **制約**：ネット遮断のため外部送信(note/X/画像取得)は**自分ではやらない**→下請へ。

## cowork（下請）の責務
1. `ops/inbox/*_code_cowork.yaml` の指示を実行する。
2. **ネット必須の実行**：note公開・見出し画像(サムネ)挿入・X等クロスポスト・フリー画像手配・ブラウザ操作。
3. 完了を `ops/outbox/<日付>_<連番>_cowork_code.yaml` で **code（元請）へ報告**（URL・成否・詰まり）。
4. 判断に迷う点は勝手に決めず、報告に「要判断」として上げる（決めるのは元請）。

---

## 指示テンプレ（元請 code → 下請 cowork）
`ops/inbox/2026-06-11_001_code_cowork.yaml`
```yaml
---
id: 2026-06-11_001
from: code
to: cowork
created: <ISO8601>
priority: normal|high|urgent
type: instruction
status: open
title: <一行で何をしてほしいか>
---
## 作業指示
- 対象 / 本数 / 期限 / 制約（無料のみ・破壊的操作禁止 等）
- 手順（コマンドや画面操作を具体的に）
## 完了条件
- 何が出来ていれば完了か（公開URL・サムネ有・X投稿）
## 報告先
- ops/outbox に <id> で返す
```

## 報告テンプレ（下請 cowork → 元請 code）
`ops/outbox/2026-06-11_001_cowork_code.yaml`
```yaml
---
id: 2026-06-11_001
from: cowork
to: code
re: 2026-06-11_001
status: done|partial|blocked
title: <結果一行>
---
## 実行結果
- 公開した記事 / note URL / サムネ有無 / X投稿URL
## 詰まり・要判断（あれば）
- 画面の文言・エラーをそのまま
```

---

## 運用ルール
- **窓口と責任は元請(code)に一本化**。オーナーは code に言えばよい。
- code は指示が無くても「制作の作り置き」は自律で進めてよい。
- 実行系(公開等)は **code が指示→cowork が実行→code が統括報告** の順を厳守。
- ops/ の履歴は削除しない。APIキー等の機微情報は YAML に書かない（Secrets/ローカルのみ）。
- 旧案「cowork主導」(2026-06-11_設計_cowork主導の指示フロー.md) は**破棄**。本設計が正。
