# 日次改善ログ テンプレート

**用途**：スタンドアップで気づいた改善ポイントを蓄積し、月次・四半期で振り返りに使う。
**運用**：1日の終わりに5分で記入。記入しない日があってもOK（過剰な負荷をかけない）。
**保存先**：`CDO/research/improvements/YYYY-MM-DD_improvement.md`
**費用**：¥0（既存テキストエディタのみ）

---

# テンプレート（コピペ用）

```markdown
# 改善ログ：YYYY-MM-DD（曜日）

## 1. 今日の作業から見えた「無駄・摩擦」

具体的に書く（抽象的な「効率化したい」ではなく、何が・なぜ・どれくらい）。

例：
- note 記事公開時、Vol.1リンクを差し替えるのに毎回5分かかる → 集客記事スキャフォルドに販売URL欄を組み込めば省略可能
- クラウドワークスで案件評価する時、複数タブを行き来するのが面倒 → 評価チェックリストの並び順を見直す

## 2. 改善案（小さく実行可能なもの）

完璧な改善ではなく、「今週中に試せる小さな1歩」を書く。

例：
- スキャフォルド JSON に sales_url フィールドを追加（30分作業）
- 評価チェックリストの上から「単価」「発注実績」を最優先に再配置（10分作業）

## 3. 実験：試したいアイデア

**仮説**と**検証方法**を書く。

例：
- 仮説：note 記事のタイトルを「〜方法」より「〜習慣」にするとPVが上がる
- 検証：次の記事で「方法」「習慣」両パターンを A/B テスト（30日後比較）

## 4. 削除・撤退の判断

「今やってる作業のうち、やめても支障ないもの」を書く。

例：
- Vol.1〜3 の集客記事に対して X スレッド5投稿を準備したが、まだ note 公開すらしていないので X 運用は1ヶ月後に保留
- 業種別サンプル記事の追加生成は、最初の1本で受注が出るまで待つ

## 5. 学び（成功・失敗の両方）

例：
- 学び：CFO の.gitignore 設定で雛形と実データを分離する設計は正解だった
- 失敗：Vol.4 が既に設計済みだったのに気づかず重複しそうになった → 設計前に既存ファイルを必ず確認するルール化

## 6. 次の改善サイクル（明日／明後日に試すこと）

最大3つに絞る。

1.
2.
3.
```

# 改善ログを書くコツ

## ① 抽象論を避ける

❌ 「もっと効率的に作業したい」
✅ 「Vol.5 設計に集中していたら、応募作業の時間が0になった。応募は1日5件のリマインダーが必要」

## ② 小さい改善を書く

❌ 「note 集客の戦略を全面的に見直す」
✅ 「集客記事のタイトル末尾に【保存版】を入れて、保存数の変化を見る」

## ③ 数字を意識する

❌ 「今日は良い日だった」
✅ 「今日はコミット5本・行数+800、応募0件、note公開0件」

## ④ 失敗を残す

成功だけでなく「やってみてダメだった」も残す。
同じ失敗を翌週繰り返さないため。

# 月次振り返りでの使い方

月初に過去30日分の improvement_log を一括レビューし、下記を集計：

1. **頻出する摩擦** TOP3 → CDO で恒久的な改善（ツール化／プロンプト化）
2. **検証中の仮説** → 結果を集計して次月の方針に反映
3. **撤退判断したもの** → 翌月のスコープから完全除外
4. **学び** → CDO/research/learnings.md に転記
