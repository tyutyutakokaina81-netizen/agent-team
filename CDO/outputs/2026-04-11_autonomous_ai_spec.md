# 自立型AI "Always-On Company" 設計仕様書

**起案日**: 2026-04-11
**起案**: CDO
**バージョン**: v0.1（構築開始版）
**関連ブリーフ**: `projects/2026-04-08_月30万自動化/brief.md`

---

## 1. ミッション定義

5役職AI（CDO/CFO/CMO/CPO/CSO）が **日次ループで自律稼働** する常駐型AI企業を構築する。

- オーナーの作業介入：**ゼロ**
- 収益目標：**4月はゼロ**（基盤構築月）
- 最終到達像：法制度・AI agent wallet・プラットフォームAPI が追いついた瞬間、既に完成された仕組みで待機する状態

---

## 2. 設計原則

1. **再現性最優先** — 誰が起動しても同じ動作をする
2. **シンプル優先** — 派手な機能より、確実に動く最小構成
3. **金銭封じ込め** — 1日・1週間・1月の支出 hard limit を多重で持つ
4. **人間介入ゼロ前提** — 承認フローは optional、未承認のまま動き続けても破綻しない設計
5. **観測可能性** — 全ての意思決定・支出・成果物が Git 履歴から追跡可能
6. **段階的立ち上げ** — CDO 単独ループ → 5役職対応 → 連続稼働、の順

---

## 4. コンポーネント仕様

### 4.1 `budget_guard.mjs`（最優先）

**責任**: Claude API 支出の封じ込め。暴走防止の第一防衛線。

**データ**:
- `CFO/research/daily_spend.json` — 日次支出記録
- `CFO/research/monthly_spend.json` — 月次支出記録
- `CFO/outputs/budget_config.json` — 上限設定

**上限ルール**:
- 1日 ¥100（Claude API 概算、Haiku 中心）
- 1月 ¥2,000
- 1ループあたり ¥30（1ループ=5役職×1ターン）

**インターフェース**:
```javascript
// 呼び出し前チェック
const ok = await budgetGuard.canProceed({ estimate: 30 });
if (!ok.allowed) {
  console.error("Budget blocked:", ok.reason);
  process.exit(0); // 正常終了（失敗扱いしない）
}

// 呼び出し後記録
await budgetGuard.record({
  officer: "CDO",
  model: "claude-haiku-4-5-20251001",
  inputTokens: 1200,
  outputTokens: 800,
  estimatedYen: 12
});
```

### 4.2 `orchestrator.mjs`

**責任**: 1ループ＝5役職のターン実行を順序管理する。

**ループ定義**:
```
loop = [
  CDO_turn,   // 自分たちの仕組みをメンテ
  CPO_turn,   // プロダクト素材を1本
  CMO_turn,   // 種まき資産を1本
  CSO_turn,   // 顧客シミュレーション・営業素材
  CFO_turn    // 支出集計・レポート
]
```

**エラー方針**: 1役職失敗しても他役職は続行。失敗は `CDO/research/errors.log` に蓄積。

### 4.3 `officer_runner.mjs`

**責任**: 1役職ぶんのターンを完結させる関数。

**入力**:
- 役職名（"CDO"等）
- 直近のコンテキスト（context/ のファイル、_index.md）
- 他役職からのメッセージ（task_queue）

**出力**:
- research/ または outputs/ に新ファイル
- _index.md に追記
- task_queue に他役職向けメモ（あれば）

**Claude 呼び出しテンプレート**:
```
System: {役職の prompt.md}

Context:
- 今日の日付
- 直近のコミット5件の要約
- 他役職の最新 _index.md 抜粋
- 自分の _index.md 直近3件

Task:
今日のあなたの1タスクを決めて実行してください。
以下のフォーマットで出力してください：
1. 今日やること（1〜2行）
2. 成果物の種類（research/ or outputs/）
3. ファイル名
4. 本文
5. _index.md への追記行
```

### 4.4 `context_loader.mjs`

**責任**: context/ ＋ 他役職の _index.md を集約して各役職に渡す。

### 4.5 `memory.json`

**責任**: 各役職の短期記憶（直近5ターン要約）と長期記憶（月次の総括）。Git で履歴管理。

### 4.6 `task_queue.json`

**責任**: 役職間の非同期メッセージング。

```json
{
  "queue": [
    {
      "from": "CSO",
      "to": "CPO",
      "priority": "normal",
      "created": "2026-04-15T10:23:00Z",
      "message": "顧客ヒアリング想定で、飲食店向けFAQ 3項目が欲しい"
    }
  ]
}
```

### 4.7 Review Dashboard（`pipeline_server.mjs` 拡張）

**責任**: オーナーが週末にスマホで覗くダッシュボード。

**エンドポイント追加**:
- `GET /dashboard/weekly` — 今週の成果物一覧・支出サマリ
- `GET /dashboard/queue` — 承認待ちキュー（あれば）
- `POST /dashboard/approve` — キュー項目の承認/却下

**承認ゼロでも動く**：デフォルトでは AI が自動確定し、オーナーは事後観察のみ。

## 6. 失敗シナリオと対処

| シナリオ | 対処 |
|---------|------|
| Claude API が上限超過 | budget_guard が即停止、翌日まで待機 |
| 役職が Claude 応答を得られない | そのターンはスキップ、errors.log に記録 |
| Git push が失敗 | 3回リトライ、それでも失敗なら翌ループで再試行 |
| 役職が無限ループに陥る | 1ターン = 1成果物 = 1commit の制約で物理的に防止 |
| 成果物が低品質で蓄積 | CDO が週1回 meta review を実行、プロンプト改善 |
| context/ が空のまま | デフォルトで company.md と CLAUDE.md を読み込む |

## 7. オーナーの理想的な関わり方（任意）

- **平日**: 何もしない。動いていることを忘れて良い
- **週末（任意）**: ダッシュボードで今週の成果物を眺める（10分以内、やらなくても良い）
- **月末**: Claude API 請求を確認。上限内に収まっていることを確認
- **将来（「接続点」を開く決断をした時）**: 蓄積された素材の中から何を最初に出すか、1つだけ決める

---

## 8. 非目標（やらないこと）

- ❌ 外部プラットフォームへの自動投稿
- ❌ 顧客との直接接触
- ❌ 決済の自動処理
- ❌ 新規アカウントの自動開設
- ❌ 本人情報の外部露出
- ❌ 4月中の収益目標追求
