# 完全自動化パイプライン設計（2026-05-05）

**目的**：テンプレ販売の「企画 → 設計 → ビルド → 公開 → 告知 → 集計 → 改善」を最大限自動化。
**前提**：オーナーは「最終アップ承認」のみ。Claude が事前準備をすべて完了させる。

---

## 1. 自動化スコープマップ（現状 vs 目標）

| 工程 | 現状 | 目標 | 自動化主体 |
|------|------|------|-----------|
| ① 企画（テーマ選定・市場調査） | 手動 | Claude が context/ideas/ から候補抽出→提案 | Claude |
| ② 設計書作成 | Claude | Claude（テンプレ化） | Claude |
| ③ 販売ページ文作成 | Claude | Claude（型化済） | Claude |
| ④ .xlsx 実体ビルド | Claude（本日達成） | Claude（汎用テンプレタ化） | Claude |
| ⑤ Drive→Sheets 変換 | 手動 | 半自動（CLI＋手動承認） | オーナー |
| ⑥ note/BOOTH 登録 | 半自動（publish.py） | 同 | オーナー |
| ⑦ SNS告知 | 半自動（publish.py） | 同 | オーナー |
| ⑧ 売上記録 | 手入力 | 半自動（CSV取込） | オーナー＋Claude |
| ⑨ 集計・ダッシュボード更新 | 手動 | 完全自動（launchd 週次） | Claude |
| ⑩ 改善提案 | 手動 | 月次レビュー自動生成 | Claude |

**ボトルネック**：⑤・⑥・⑦ はプラットフォームAPI制約で完全自動化不能。最良は publish.py パターン。
**自動化伸びしろ**：①・④・⑧・⑨・⑩ は Claude 完全自動化が可能。

## 2. 残工程の自動化候補（ROI順）

### 優先度A（今月中に着手すべき）

#### A1. Vol.N 汎用テンプレタ化（`scripts/build_template.py`）
- **入力**：YAML 設定ファイル（テーマ・列構成・数式・テーマリスト）
- **出力**：完成版 .xlsx + マニュアル PDF + note/BOOTH 販売ページ md
- **削減効果**：Vol.5 以降 の制作時間が「設計2h → 実体生成5min」に
- **依存**：openpyxl・docx・pdf スキルすべて
- **実装工数**：4時間（1日1テーマ）

#### A2. 売上 CSV 自動取込（`scripts/import_sales.py`）
- note 管理画面・BOOTH管理画面からエクスポートした CSV を、
  CFO台帳の販売台帳シートに自動マージ
- **削減効果**：月1回の手入力 5分 → ファイル投入 30秒
- **実装工数**：1.5時間

#### A3. ダッシュボード週次自動更新（既設計済）
- `2026-05-05_dashboard_週次自動更新設計.md` の通りに実装
- launchd 日曜23:00 で自動実行
- **削減効果**：週次の手動更新 15分 → ゼロ
- **実装工数**：30分（既設計あり）

### 優先度B（次月以降）

#### B1. 月次経営レビュー自動生成（`scripts/monthly_review.py`）
- ダッシュボード・売上CSV・進捗ログを統合し、月初に「先月のレビュー＋今月の優先タスク」を Markdown で自動生成
- **削減効果**：月次レビュー資料作成 1時間 → ゼロ
- **実装工数**：2時間

#### B2. 案件発見の自動化（柱A拡張）
- 既存 `pipeline/` パイプラインの再活性化（Google OAuth 解決後）
- CrowdWorks/ランサーズの新着案件を Playwright でスクレイプ→ Claude で評価→ 応募候補を `apply.py` 形式で出力
- **削減効果**：手動の案件探索 30分/日 → ゼロ
- **実装工数**：4時間（既存資産流用）

### 優先度C（事業規模拡大時）

#### C1. CSO 顧客対話の半自動化（FAQ Bot）
- 購入者からのコメント・DM への自動下書き提案
- Claude を CLI 経由で呼び、定型回答パターンと文脈の組み合わせを生成
- **実装工数**：3時間

#### C2. SNS 投稿スケジューラ（Buffer/Hootsuite等の代替）
- `publish.py` を `cron`/`launchd` で時刻指定実行
- 即時投稿のみ（API制約で予約投稿は不可）
- **実装工数**：1時間

## 5. Vol.N 汎用テンプレタ仕様（A1の詳細）

### 入力（YAML 設定例）

```yaml
vol: 5
title: "ブログ記事構成テンプレ"
price: 980
target_audience: "個人ブロガー・コンテンツマーケター"
sheets:
  - name: 構成テンプレート
    columns: [No., 記事タイトル, 想定読者, 検索意図, 見出し1, 見出し2, 見出し3, CTA]
    rows: 30
  - name: チェックリスト
    columns: [No., 項目, 完了]
    rows: 20
manual:
  - "STEP1: ファイルをコピーして使う"
  - "STEP2: 記事案を構成テンプレートに記入"
sales_copy:
  pain_points:
    - "ブログを書こうとして手が止まる"
    - "記事の構成に時間がかかる"
  features:
    - "30個の記事構成テンプレ"
    - "チェックリストで品質担保"
```

### 出力

- `Vol5_ブログ記事構成テンプレ.xlsx`
- `vol5_note_listing.md`（販売ページ文）
- `vol5_launch_pack.md`（ローンチパック・告知文含む）
- `publish.py` への新タスク追記（10〜20番として）

### コア処理

1. YAML 読込・バリデーション
2. openpyxl で .xlsx 生成（既存 build_vol2.py のロジックを汎用化）
3. テンプレート md に変数を流し込み（Jinja2 等）
4. recalc.py で数式検証
5. _index.md に成果物追記

## 6. Claude起動時の自動セットアップ（rule 13 強化）

各セッション冒頭で Claude が自動実行すべき30秒ルーティン：

```
1. AGENT_RULES.md を Read（v1 13項目を再確認）
2. CLAUDE.md を Read（プロジェクト全体ルール）
3. projects/2026-04-08_月30万自動化/PLAYBOOK.md を Read（事業全体）
4. dashboard.md を Read（最新進捗）
5. 各役職の _index.md「進行中タスク」を grep（残タスク把握）
```

これを `START_HERE.md` または `CLAUDE.md` に明記すれば、毎回の Claude が同じスタートラインに立てる。
