## 目的

Claude Code から Gmail を直接操作し、以下を一貫処理できるか確認する：
1. 受信メールの読み取り
2. 返信文の起案
3. PDF（請求書・見積書など）の生成と添付
4. 送信（または下書き保存）

---

## 動画から読み取れた構成（要検証）

杉田氏は **「gemini-cli」（音声では「GOG」と発音）** を介して Claude Code から Gmail を操作している。
ただし「gemini-cli」は固有名詞として複数解釈可能：

| 候補 | 概要 | 当社で使えるか |
|------|------|---------------|
| Google 公式 Gemini CLI | Google製の Gemini API クライアント | 直接 Gmail 操作機能は限定的 |
| MCP（Model Context Protocol）の Gmail サーバー | Claude Code 公式の連携手段 | **最も可能性が高い**（要確認） |
| 杉田氏の自社製 CLI ラッパー | 独自実装 | 入手不可 |

→ **試作段階の方針**: まず MCP の Gmail サーバー実装が公開されているか調査することから始める。

## API 費用に関する注意

| 要素 | コスト発生 |
|------|----------|
| Gmail API（読み書き） | **無料枠あり**（1日10億クォータユニット、通常使用で超過しない） |
| Google OAuth 2.0 認証 | 無料 |
| Claude Code 側のトークン消費 | **発生する**（受信メール本文を Claude に読ませる場合、本文サイズ分のトークンを消費） |

**コスト最小化策**:
- メール本文を全文ではなく要約だけ Claude に渡す
- 添付ファイル PDF はローカル生成のみ（PDF をテキスト化して Claude に渡さない）
- 一括バッチ処理ではなく、ユーザー指示時のみ実行する

## 試作の進め方（最小ステップ）

1. **調査**: Claude Code MCP マーケットプレイスで「gmail」を検索し、公式または評価の高いサーバーがあるか確認
2. **手動セットアップ**: 当該 MCP サーバーをローカルに導入し、OAuth で当社の Gmail アカウントを許可
3. **最小動作確認**: Claude Code から「直近の未読メール件名を3件表示して」と指示して動作確認
4. **PDF 添付テスト**: ダミー請求書 PDF を生成し、自分宛にメール送付して受信確認
5. **撤退基準**: ステップ3で15分以内に動かなければ撤退、別の手段を検討

## リスク（事前に把握）

| リスク | 対策 |
|--------|------|
| Gmail OAuth トークンの漏洩 | トークンは `~/.config/...` 等にローカル保存。Git にコミットしない |
| 誤送信 | 送信前に必ず「下書き保存」のみとし、ユーザー目視確認後に送信ボタンを押す運用にする |
| 顧客 PII の Claude 学習 | 本文をそのまま渡さず、件名・要約のみを渡す。クライアント情報はマスキング |
| 動画情報の精度不足 | 「gemini-cli」の正体が不明なため、調査が空振りに終わる可能性あり |

## 次にオーナーが取るべきアクション

サンドボックスからは外部 OAuth フロー・Gmail API を実行できないため、
**実機での試作はオーナー自身が手元の PC で行う必要がある**。

候補手順：
1. `claude mcp list` で利用可能な MCP サーバー一覧を確認
2. 「gmail」を含むものがあれば `claude mcp add` で追加
3. なければ、PyPI/npm で `gmail mcp` を検索して導入
4. 動作確認は本ファイルの「試作の進め方」ステップに従う
