## 1. 目的の再確認

Claude Code から Gmail を直接操作し、**請求書送付・営業メール送信・受信メールの一次仕分け**を自動化したい。当社の収益柱A（SEOライティング代行 ¥15K/記事 × 20本）・柱B（SNS運用代行 ¥50K/社 × 6社）が立ち上がると、見積→請求→入金確認→お礼までの定型メール往復が月数十件発生する。ここを Claude Code で半自動化できれば、CSO/CFO の手間が大幅に減る。動画事例（杉田氏）で言及された「gemini-cli/GOG」の正体が不明なため、**現実的に取り得る選択肢を網羅して、最短ルートを選ぶ**ことが本ドキュメントの狙い。

## 2. 候補A: 公式 MCP Gmail サーバー（Anthropic 純正）

### 把握範囲（【未検証】）
- Anthropic は Claude Code 向けに MCP（Model Context Protocol）の公式コネクタを複数提供している【推定】
- ただし Gmail を**公式純正**でカバーしているかはトレーニング知識では断定できない【未検証】
- Anthropic 公式の "connectors" には Google Drive / Google Calendar は含まれていた記憶があるが、Gmail 単体は記憶にない【要実機確認】

### 仮定的インストールコマンド（形式の参考）
```bash
# 【仮定形・実コマンドではない】
claude mcp list                          # 既存サーバー確認
claude mcp add gmail                     # 公式があればこの形式
claude mcp add @anthropic/mcp-gmail      # npm パッケージ指定形式
```

### 評価
- **存在すれば最有力**（OAuth・スコープ管理・エラー処理が枯れている前提）
- 存在しない可能性も五分五分。期待しすぎないこと。

## 3. 候補B: コミュニティ製 MCP Gmail サーバー（npm / PyPI）

### 一般的検索キーワード【推定】
- `mcp-gmail` / `gmail-mcp` / `mcp-server-gmail`
- `@modelcontextprotocol/server-gmail`（公式リポジトリ命名規則に準じた仮名）
- GitHub topic：`mcp`, `gmail`, `claude`

### 評価基準（実機で選ぶときのチェックリスト）
| 観点 | 基準 |
|------|------|
| OAuth 対応 | OAuth 2.0 / refresh token 対応必須。App password は不可（Google 廃止傾向） |
| 読み書き両対応 | `gmail.readonly` だけのものは要件不足。`gmail.send` / `gmail.modify` 必須 |
| 最終更新日 | 3ヶ月以内に更新があるか（MCP 仕様自体が変動中のため） |
| Star 数 / Issue 解決速度 | 100★以上、Issue 1週間以内に反応 |
| ライセンス | MIT / Apache-2.0 が望ましい |
| 添付ファイル対応 | PDF 添付・MIME multipart 送信が API で叩けるか |

### 現実的な見立て
**実務ではこの候補Bが本命になる可能性が高い**。MCP エコシステムはコミュニティドリブンで、Gmail のような定番サービスは複数の実装が並立している状況【推定】。複数候補から「最終更新が直近」「Issue が静かに閉じている」ものを選ぶ。

## 4. 候補C: 自作シェルスクリプト（Gmail API 直叩き）

MCP に拘らず、`curl` + Gmail REST API を Bash で叩いて Claude Code から `Bash` ツール経由で呼び出す方式。**最も枯れていて壊れにくい**が、初期構築コストは高い。

### Google Cloud Console 側の準備（高レベル）
1. GCP プロジェクトを新規作成（例：`agent-team-gmail`）
2. Gmail API を有効化
3. OAuth 同意画面を「外部・テストモード」で作成、テストユーザーに自分の Gmail を追加
4. 認証情報 → OAuth 2.0 クライアント ID を「デスクトップアプリ」で作成
5. `credentials.json` をダウンロード → **Git にコミット禁止**

### 必要なスコープ
| スコープ | 用途 |
|---------|------|
| `https://www.googleapis.com/auth/gmail.readonly` | 受信メール読み取り |
| `https://www.googleapis.com/auth/gmail.send` | 送信 |
| `https://www.googleapis.com/auth/gmail.modify` | 既読化・ラベル付け・下書き保存 |
| （`gmail.compose` は send+下書きのサブセット、別途要否判断） |  |

### 認証情報の保存場所
```
~/.config/agent-team/gmail/credentials.json   # OAuth クライアント
~/.config/agent-team/gmail/token.json         # アクセストークン・refresh token
```
- パーミッション `600` 必須
- `.gitignore` に `**/credentials.json`・`**/token.json` を追加
- リポジトリ内（`CDO/outputs/` 等）には**絶対に置かない**

### 評価
- 依存ゼロに近く、サーバーが落ちたら原因が即特定できる
- 反面、refresh token の更新・エラーハンドリング・MIME multipart 構築を自前で書く必要があり、最小実装でも 200 行は超える【推定】
- 候補A/Bが動かなかった場合の**最終フォールバック**

## 5. 候補D: Google 提供 "Gemini CLI"（動画の "GOG" 候補）

### 把握範囲（【未検証】）
- Google は Gemini API クライアントとして `gcloud` 系または独立 CLI を提供している記憶がある【推定】
- 名称候補：`gemini` CLI / `google-genai` SDK 付属の CLI / Google AI Studio 連携ツール
- **これ単体で Gmail を操作できるかは記憶にない**。Gemini はあくまで LLM 推論 API のクライアントであり、Gmail 操作は Workspace 側の権限が別途必要【推定】
- 動画の「GOG」が **Google 公式 Gemini CLI** を指していたとしても、その CLI が裏で MCP 的に Gmail を叩いていたとは限らない。**杉田氏が独自にラップしていた可能性もある**

### 評価
- 当社が Claude Code を主軸に置いている以上、Gemini CLI をわざわざ挟むメリットは薄い
- 動画事例の再現性を確認する**参考調査**としてのみ価値あり。本命候補ではない
