## 1. 2026年プロンプト設計の根本原則

### 1-1. コア原則：明示性・構造・成功基準
- Claude は **明確な成功基準・構造化された入力・明示的な出力制約**で最大性能を発揮する
- 推論に頼らず「契約書」のように厳密に書くと結果が安定する

### 1-2. 「契約書スタイル」のシステムプロンプト
```xml
<role>あなたは〇〇の専門家</role>
<success_criteria>
- 基準1：...
- 基準2：...
</success_criteria>
<constraints>
- 禁止事項
- 必須事項
</constraints>
<uncertainty_handling>
不確実な場合は推測せず「分かりません」と明示
</uncertainty_handling>
<output_format>
- セクション1
- セクション2
</output_format>
```

### 1-3. XMLタグの活用（2026年の決定打）
Claude は XML タグを認識するよう訓練されており、**markdown や区切り文字よりも信頼性が劇的に向上**する。

```xml
<task>...</task>
<context>...</context>
<examples>
  <example>入力 → 出力</example>
</examples>
<instructions>...</instructions>
```

## 2. プロンプトの最適な長さ・構造

| 項目 | 推奨 |
|------|------|
| プロンプト長 | **150〜300語**（多くのタスク） |
| 重要情報の配置 | プロンプトの **先頭10% と 末尾10%** に配置 |
| 中盤の情報 | 注意機構の偏りで「Lost in the Middle」が発生 |

**実装テクニック**：長文プロンプトでは「重要指示→詳細→重要指示の再掲」のサンドイッチ構造。

## 3. 効果が高いテクニック

### 3-1. Few-Shot Examples（最強の技）
2〜3例の入出力ペアを示すことで、フォーマット一致率が飛躍的に向上。

```xml
<examples>
  <example>
    入力：[例1の入力]
    出力：[例1の出力]
  </example>
  <example>
    入力：[例2の入力]
    出力：[例2の出力]
  </example>
</examples>
```

### 3-2. 不確実性の許可
「分からない時は推測せず『不明』と答えてください」を必ず入れる → ハルシネーション削減。

### 3-3. Chain of Thought（思考連鎖）
複雑な推論タスクでは `<thinking>` タグで段階的思考を促す。

## 4. 月30万自動化プロジェクトへの適用

### 4-1. 柱A（SEOライティング）プロンプトの強化

**現状**：`A_ライティング/service_page.md` のプロンプトテンプレは [...] プレースホルダ式

**強化案**：
```xml
<role>SEOライターとして、検索意図に応える記事を生成</role>

<success_criteria>
- メインKWを冒頭100字以内に含める
- H2見出し4〜6個、各H2にH3を2〜3個
- E-E-A-T観点で「経験・専門性」を1段落以上含む
- 文字数±10%以内
</success_criteria>

<inputs>
  <keyword>{KW}</keyword>
  <persona>{ペルソナ}</persona>
  <wordcount>{文字数}</wordcount>
  <tone>{トーン}</tone>
</inputs>

<constraints>
- 根拠不明な断言禁止
- 同一文末表現の3回以上連続禁止
- AI典型語（「revolutionize」「seamless」等）禁止
</constraints>

<output_format>
1. タイトル（32字以内）
2. リード文（300字）
3. 目次
4. 本文
5. まとめ（300字）
6. メタディスクリプション（120字以内）
</output_format>
```

### 4-2. 柱B（SNS投稿生成）の強化
- 月初に「クライアントカード」をXMLで構造化 → 毎回の投稿生成で再利用
- 「Brand Voice Examples」を Few-Shot で3例提示 → トーン一貫性が向上

### 4-3. 柱C（テンプレ販売文）の最適化
- 販売ページ文は「PASTOR（Problem→Amplify→Story→Transformation→Offer→Response）」テンプレを XML 化
- A/Bテスト用にバリエーション生成プロンプトを用意

### 4-4. 柱D（応募文生成）の品質改善
- 既存パイプラインは Few-Shot 強化で応募通過率向上余地あり
- 過去成功した応募文を `templates/successful_applications.xml` に蓄積し参照させる

## 5. エージェント型アーキテクチャ（2026年の潮流）

### パラダイムシフト
- 2024年：個別リクエストの最適化
- **2026年：システム全体の設計**（半自律的に動くワークフロー）

### 月30万プロジェクトへの応用
```
柱D パイプラインの拡張：
01_search → 02_evaluate → 03_apply → 04_execute → 05_review → 06_deliver

各ステップを Claude エージェントとして設計：
- 各エージェントに「契約書スタイル」の明確な役割と成功基準
- ステップ間のデータ受け渡しは JSON で構造化
- 失敗時のエスカレーション基準を明示
```

## 6. Claude Code 活用テクニック

### CLAUDE.md の最適化
- プロジェクトルートに置き、Claude が自動読込
- 「絶対ルール」「ファイル命名」「禁止操作」を明示
- 既に本リポジトリは良い構造（強化余地は少ない）

### サブエージェント活用
- `.claude/agents/` にカスタムエージェント定義
- 例：「sales-writer」「proofreader」「code-reviewer」

### MCP（Model Context Protocol）
- 外部ツール連携（Notion、Google Drive、Sheets）
- 月30万プロジェクトでは Google Sheets MCP で売上自動集計が実現可能

## 7. セキュリティと規約遵守

### プラットフォーム規約の現状（2026年）
- クラウドワークス／ランサーズ：AI使用は**明示が必須**（透明性原則）
- 「AI生成」ラベルを付与する案件は受注可、付与しない隠匿はNG

### Prompt Injection 対策
- ユーザー入力は必ず `<user_input>` タグで囲み、システム指示と分離
- 外部URLのコンテンツを Claude に渡す前にサニタイズ
