## 3. 推奨処置（公開濃厚14本＋全体）

1. **公開前 note 検索の必須ゲート化（全21本・特に濃厚14本）**：
   - 新タイトルの検索だけでは不十分（note上のタイトルはファイル名と一致しない実例多数）。
   - **題材キーワードで検索**すること：「げんげ」「勝興寺」「海王丸」「御車山」「二上山」「沖漬け」「山町筋」「岩がき/岩牡蠣」「富山城」「トロッコ/黒部峡谷」「岩瀬」「赤巻き」「水羊羹」「バイ貝」。
   - **0件＝公開GO／1件以上＝公開中止して code に報告**（氷見牛方式＝キュー記事は公開禁止化し、既存記事へのフッター/加筆で統合するか題材を差し替える）。
2. **氷見線・山町筋の2本は先に ops/inbox/2026-07-01_005【0】の実行結果を確認**してから扱う（実行済みなら重複、未実行なら通常公開）。URL報告が無い限り公開済み扱いにしない。
3. **published_registry.tsv の全件化を公開再開の前提条件（ブロッキング）にする**：
   cowork の note API で全330件の note ID＋タイトルを取得（発注済：ops/processed/2026-06-30_001【1】）。
   これが揃えば本監査の「濃厚」は全て白黒確定できる。`inject_js/` の34 ID の題材特定も同時に行う。
4. **「URL報告なき公開」の禁止を恒久ルール化**：公開＝outboxにURL必須。URLの無い公開報告（「公開5件」だけ等）が
   今回の判定不能（6/11の5本どれか不明 等）の直接原因。
5. **ブランチ残留記事の棚卸し**：brave-morse／wizardly-lovelace／vibrant-keller／new-session 等に残る
   未マージ記事は「公開されたかもしれない原稿」として registry 照合が済むまで**同題材の新規執筆を禁止**する
  （今回の21本中14本がこのパターンで濃厚判定になった）。
6. **軽微な修正**：`CMO/outputs/2026-07-01_配信キット_X投稿＋分散ルーティン.md` のX投稿1番

## 4. 事実と意見の区分

- 【事実】上記1章のURL30件・報告済み公開（6/9の25本GO文言・6/10の5本・6/23の10本 等）は repo 内ファイルに明記されている。
- 【考察】「公開濃厚」14本は、①同題材の旧記事が一斉公開期（6/9〜6/12・note404本・未来日付公開あり）のツリーに存在した、②公開報告が題名レベルで残っていない、という状況証拠による推定。紅ずわいがに（7/01に0件確認→公開）の反例があるため断定はしない。
- 【提案】3章の通り。特に「題材キーワード検索0件確認」と「registry全件化の先行」の2点をゲートにすれば、氷見牛型の事故は構造的に再発しない。
