# Webツール群を「広告/フリーミアムで稼ぐメディア資産」にする戦略

作成: 2026-06-23 ／ 担当: CMO 兼 CAO ／ 区分: 別モデル（コンテンツ販売とは別の収益線）

---

## 0. 要点（結論先出し）

- **モデルの定義**：無料Webツール（採算計算機・請求書ジェネレータ・プロンプトビルダー等）を作り、
  **検索流入（SEO）で集客 → ①ディスプレイ広告 ②アフィリエイト ③フリーミアム（自社商品/Proへ送客）** の3線で収益化する。
  これは「記事や教材を売る」モデルとは別物で、**売り物はツールではなく“集まったトラフィック”**。
- **ゼロ予算で成立**：ツール本体は静的サイト（HTML+JS）でGitHub Pagesに無料ホスト。サーバ不要・課金なし。
  集客は有料広告を一切使わず、SEOと無料SNS/Redditのみ。
- **正直な見立て**：このモデルは**「速くない」**。広告は審査とPVのハードルがあり**最初の数ヶ月はほぼ¥0**。
  SEOで検索順位がつくまで**3〜6ヶ月**かかる。短期の現金にはならない。**本命=半不労の資産形成**。
  即金が要るなら受託（既存3本柱①）が先で、本モデルは並行で“仕込む”位置づけ。
- **本会社の確定方針との整合**：STATE 2026-06-22「ゼロ予算・無料ツールのみ・広告は来たら拾う」と完全に一致。
  **本モデルは“広告営業をしない受け身の広告収益（AdSense等の自動配信）”なので、方針内**。
- **A制約順守**：効果や金額は**断定せず必ず「目安・条件つき」**で書く（A5誇張NG）。PII不記載（A4）。

## 1. 収益モデル（3本立て・現実的な見立て）

ツール系サイトの収益は基本この3つの組み合わせ。1本に賭けず、**並走させて取れるものを取る**。

### (a) ディスプレイ広告（Google AdSense 等）

ページに広告枠を置き、表示・クリックで自動的に収益が入る方式。**営業不要・受け身**なので本会社方針に合う。

**条件（正直に）**
- **審査がある**：AdSenseはサイト審査を通らないと1円も入らない。一般に通りやすいのは
  「独自性のあるコンテンツ／プライバシーポリシー・運営者情報あり／十分なページ数と中身」。
  **ツール単体（1ページぺら）だけだと審査落ちしやすい** → 各ツールに**解説テキスト/使い方/FAQを併設**して“中身のあるページ”にする必要がある。
- **PVが収益を決める**：広告は **「PV × 表示単価」**。日本語サイトの一般的な目安（断定不可・あくまで桁感）は
  **RPM（1000PVあたり収益）≒ ¥100〜¥500**程度に収まることが多い。ジャンル・季節・広告主で大きく振れる。
  - 桁感の例（**保証ではない目安**）：
    - 月 **1万PV** → おおよそ **¥1,000〜¥5,000/月**
    - 月 **10万PV** → おおよそ **¥1〜5万/月**
    - 月 **50万PV** → おおよそ **¥5〜25万/月**
  - → **「最初の数ヶ月はPVが少なく、ほぼ¥0」**が現実。資産が育って初めて意味のある額になる。
- **速さ**：遅い（審査＋PV育成で数ヶ月〜）。**上限**：PV次第で青天井だが、個人ツールサイトで月10万PV超は相応の本数と時間が要る。

**代替/補完**：審査が重ければ、まず**アフィリエイト型広告（A8.net・もしもアフィリエイト等の無料ASP）**から始めるのも手。
これらは審査が比較的ゆるく、PVが小さくても**成果（クリック→申込）**が出れば入る。

### (b) アフィリエイト（ツールに関連する商品・サービス）

ツールの利用文脈に**自然に合う商品/サービス**を紹介し、申込・購入で報酬を得る。**文脈一致なら成約率が高い**のが強み。

| ツール | 文脈に合うアフィリ対象（例・一般カテゴリ） | 報酬の出やすさ |
|---|---|---|
| フリーランス採算/時給計算機 | 会計ソフト、開業freee/マネーフォワード、請求書サービス、フリーランス向け保険・福利、案件紹介サービス | 高（申込報酬が大きめ） |
| 請求書ジェネレータ | 会計/請求SaaS、電子帳簿保存対応サービス、印鑑/電子契約 | 高 |
| AIプロンプトビルダー | AIツールの有料プラン、オンライン講座、書籍（Amazonアソシエイト）、ガジェット | 中（単価は低め〜中） |
| 副業/物販計算系 | クラウドソーシング登録、物販ツール、ネット銀行/カード | 中〜高 |

**条件（正直に）**
- ASP登録＝多くは**無料**。物販（Amazonアソシエイト等）は審査・規約あり。**虚偽/誇張表現は規約違反＝A5とも整合（禁止）**。
- **速さ**：広告より速く初収益が出ることがある（PVが小さくても成約すれば入る）。
- **上限**：紹介できる高単価サービスの数と、ツールの利用者数に依存。**月数千〜数万円が現実的な初期レンジ**。
- **設計の肝**：押し売りしない。「計算した後に“次の行動”として自然に必要になるもの」を1〜2個だけ置く（リンクの貼りすぎは離脱とSEO悪化を招く）。

### (c) フリーミアム（無料ツール → 有料Pro / 自社商品へ送客）

無料ツールで集客し、**①自社の既存商品（Kindle/Gumroad/テンプレ集/プロンプト集）**、または
**②ツールの上位版（Pro）**へ送る。**最も会社資産と噛み合う線**。

**送客先（既に在庫がある）**
- AIプロンプト集30本 / 実務テンプレ集8種＋CSV（`projects/2026-06-23_収益化_AIひとり会社商品/`）
- Kindle×4・英語版Gumroad（同上）
- → 例：プロンプトビルダーの出力欄の下に「もっと使えるプロンプト30本はこちら（Gumroad）」を1行だけ。

**“Pro版”の作り方（ゼロ予算で可能な範囲）**
- 無料：基本計算/単票生成。
- Pro（有料）：CSV一括・履歴保存・ロゴ入りPDF・広告非表示 等。
- 課金は**Gumroad/Stripe Payment Link等の外部決済**で受ける（自前サーバ不要）。
  ※決済導線の“実装と公開”はオーナー/cowork作業（code=設計まで）。

**条件（正直に）**
- **速さ**：送客自体はすぐ仕込めるが、**送客が効くにはツールに人が来ている前提**＝結局SEO待ち。
- **上限**：自社商品の単価×送客数。少PVでも**1人あたりの価値が広告より圧倒的に高い**（広告は1PV数十銭、商品成約は数百〜数千円）。
- **結論**：3線のうち**(c)が本命**。(a)(b)は「ついで」。**まず(c)を必ず仕込み、PVが育ったら(a)(b)が後乗りで効く**順。

## 2. 集客（ゼロ予算）— SEOが本命

ツール系は**“今まさにその作業をしたい人”が検索する**ので、検索意図がはっきりしている＝SEOと相性が最高。
有料広告は使わない。やることは**(A)検索に当てる設計 (B)ツールハブ化（相互リンク）(C)無料SNS/Redditでの初速**。

### (A) 検索意図に当てるタイトル/URL設計

- **URL**：英語スラッグで意図を直球に。短く・1ツール1ページ・恒久URL（後で変えない）。
  - 例：`/tools/freelance-hourly-rate/`（時給計算）、`/tools/invoice-generator/`、`/tools/ai-prompt-builder/`
- **title/h1**：検索語そのものを含める。「〇〇 計算」「〇〇 作成 無料」「〇〇 テンプレート」等、**実際に打たれる語**に合わせる。
- **各ページに“中身”**：ツール＋(使い方/計算式の説明/よくある質問/関連リンク)。これが**AdSense審査通過とSEO評価の両方**に効く。
- **メタ**：description は「無料・登録不要・ブラウザで完結」を明記（クリック率に効く）。

### (B) 各ツールの狙いキーワード（初期ラインナップ案）

会社資産（フリーランス/AI/ひとり会社）と検索需要が重なるところを優先。

| ツール | 狙う検索意図（語の方向性） | 送客先（フリーミアム） |
|---|---|---|
| **フリーランス時給/採算計算機** | 「フリーランス 時給 計算」「自分の時給 計算」「単価 妥当」 | 会計SaaSアフィリ＋時給記事＋実務テンプレ集 |
| **請求書ジェネレータ（無料・登録不要）** | 「請求書 作成 無料」「請求書 テンプレート」「インボイス 番号」 | 請求SaaSアフィリ＋Proで(ロゴ/CSV)＋テンプレ集 |
| **AIプロンプトビルダー** | 「ChatGPT プロンプト 作り方」「プロンプト テンプレート」 | プロンプト集30本（Gumroad）＋AI講座アフィリ |
| **副業/物販 利益計算機** | 「物販 利益 計算」「手数料 計算」 | クラウドソーシング/物販ツールアフィリ |
| **見積書ジェネレータ** | 「見積書 作成 無料」 | 請求SaaS＋テンプレ集 |

> 注：**いきなり全部作らない**。1つを検索1ページ目に入れる方が、10個を圏外で放置するより資産価値が高い（§最後の「最初に伸ばす1ツール」参照）。

### (C) 相互リンク（ツールハブ化）

- **/tools/ ハブページ**を作り、全ツールを一覧（カード型）。各ツールページのフッターから他ツールへ相互リンク。
  → 回遊が増え、1セッションのPVが伸び（広告に効く）、内部リンクでSEO評価も底上げ。
- **既存note記事との接続**：例「時給を計算する話」note → 時給計算ツール、へ相互リンク。
  既存127本のメディア資産がツールへ送客し、ツールが商品へ送客する**ファネルが一本化**される。

### (D) 無料SNS/Reddit での配布作法（初速づくり）

- SEOが効くまでの“最初の人”を入れる手段。**宣伝臭を出さず「役立つ無料ツール作った」共有**として出す。
- **X**：既存のX運用に「無料ツール作りました（登録不要・ブラウザ完結）」を時々混ぜる。スクショ1枚＋URL。
- **Reddit**：関連subreddit（例 r/freelance, r/sidehustle 等の英語圏／英語版ツールがあれば特に有効）。
  ⚠️ Redditは**自己宣伝に厳しい**。サブのルール厳守・自分の投稿割合を抑える・「便利ツール紹介」文脈で。違反すると逆効果。
  - 英語版ツールを用意できれば、Reddit/海外検索からの流入は**North Star（海外読者）とも合流**しうる。
- **note/ブログから内部送客**：これが一番安全で効く（自分の資産から自分のツールへ）。

> ⚠️ **実行はオーナー/cowork**：SNS投稿・Reddit投稿・ASP/AdSenseの審査申請やアカウント操作は外部ネット作業＝code（私）はできない。code は文面・設計・スラッグ・キーワード表まで用意する。

## 3. ホスティング（GitHub Pages・無料）

当repoは既にGitHub Pages運用中（`pages.yml` が `tavern.html` を `_site/index.html` にデプロイ）。
**同じ仕組みでツールを相乗りできる**。サーバ費・ドメイン費ゼロで開始可能。

### 構成案（既存pages.ymlを壊さず拡張）

```
_site/                       ← Pagesが配信するルート
├── index.html               ← 既存tavern（現状維持）
├── tools/
│   ├── index.html           ← /tools/ ハブ（全ツール一覧）
│   ├── freelance-hourly-rate/index.html
│   ├── invoice-generator/index.html
│   └── ai-prompt-builder/index.html
└── ...
```

- **各ツール = 単一HTML（+ インラインJS/CSS）**。ライブラリ依存最小、完全クライアントサイドで動く（計算/生成はブラウザ内）。
  → サーバ不要・データ送信なし＝**プライバシー的にも「入力は送信されません」と明記でき、信頼=コンバージョンに効く**。
- **pages.yml の Build ステップに `cp -r tools _site/tools` を1行足すだけ**で配信に乗る（ワークフローの軽微改修）。
- **ドメイン**：
  - 初期＝**無料の `<user>.github.io`** で十分（費用ゼロ・方針通り）。
  - 将来、独自ドメインが欲しくなったら**任意**（年¥1,000前後の費用が出るので、方針上は“稼ぎが出てから”）。
    独自ドメインはSEO上わずかに有利な面もあるが、**ゼロ予算原則を優先し、まずは github.io で開始**。
- **AdSense審査の前提整備（無料でできる）**：`/privacy/`（プライバシーポリシー）と `/about/`（運営者情報）の静的ページを置く。
  審査・規約上ほぼ必須。文面はcodeが用意、設置はPagesにコミットするだけ。

> code側でできること＝HTML/JS/ワークフロー差分の**作成まで**。実際の `git push`・Pages有効化・AdSense審査申請はオーナー/cowork。

## 4. 計測（無料で見るべき指標）

全部**無料ツールで足りる**。最初から完璧を狙わず、**「人が来てるか」「次へ進んでるか」**の最小4指標を見る。

| 指標 | 何で見るか（無料） | 何が分かるか |
|---|---|---|
| **PV / ユーザー数 / 流入元** | Google Analytics（GA4・無料）または Cloudflare Web Analytics（無料・軽量） | 集客が育っているか、SEOが効き始めたか |
| **検索クエリ / 表示回数 / 掲載順位** | Google Search Console（無料） | どの検索語で何位か＝SEOの本丸。**ツール系で最重要** |
| **ツール利用イベント**（計算実行・生成ボタン押下） | GA4のイベント（無料・JS数行） | 来訪者が実際に使っているか（直帰と区別） |
| **送客クリック**（商品リンク/アフィリ/Pro） | GA4イベント＋ASP管理画面＋Gumroad売上 | (c)フリーミアムが効いているか＝**収益の先行指標** |

- **見る頻度**：週1で十分（毎日は変動ノイズ）。Search Consoleの「順位が3〜4ページ目に出始めた」が**伸びの最初の兆候**。
- **判断ルール（CAO視点）**：3ヶ月見てSearch Consoleに表示すらされないツールは、タイトル/中身を作り直すか撤退。
  **少数のツールに集中**する判断材料にする。

## 5. 正直なリスク（A5：誇張しない）

- **広告は最初ほぼ稼げない**：審査通過が前提＝1ページぺらでは落ちる。通っても**PVが小さい初期は月数百円〜数千円**が現実。
  「ツール作ればすぐ広告収入」は誇張。**PVが資産として育って初めて意味のある額**になる。
- **SEOは時間がかかる**：新規ページが検索で評価されるまで**数ヶ月（目安3〜6ヶ月）**。即金にはならない。
  途中で順位が動く・Googleのアップデートで上下するなど**コントロールできない不確実性**がある。
- **競合が強い領域**：「請求書 作成 無料」等は大手SaaSが上位を占有。**ニッチ＋文脈特化（フリーランス/AI/ひとり会社）で隙間を狙う**のが現実解。真っ向勝負は避ける。
- **プラットフォーム依存**：AdSense/ASP/Redditは**規約変更・アカウント停止のリスク**。1社・1手段に依存しない（だから3線並走）。
- **メンテ負荷**：ツールは作って放置でなく、壊れ（仕様変更/法改正＝例:インボイス様式）に追従が要る。**少数精鋭**で持続可能に。
- **会社方針との線引き**：本モデルは方針内（受け身の自動広告＝OK）。ただし**「広告主に営業する」「有料広告を出す」は方針外**。やらない。

**総括**：本モデルは**“速い金”ではなく“半不労の資産”**。受託（即金）と切り分け、**並行で少しずつ仕込む**のが正しい使い方。
最悪でも**支出ゼロなので赤字化しない**（方針：赤字にしない＝続ける、に合致）。

## 6. 着手手順（ゼロ予算・最小ステップ）

1. **1ツールに絞って静的HTMLを作る**（下記の最初の1つ）。完全クライアントサイド、入力は送信しない設計。code制作可。
2. 同ページに**使い方/計算式の説明/FAQ/関連リンク**を併設（審査＆SEO対策）。code制作可。
3. `/tools/` ハブ・`/privacy/`・`/about/` を用意。`pages.yml` に `cp -r tools _site/tools`（＋静的ページ）を追加。code制作可。
4. **送客リンク（自社Gumroad商品）を1行だけ**設置（フリーミアム=(c)を最初から仕込む）。code制作可。
5. **公開**（git push / Pages）→ **GA4 / Search Console / Cloudflare Analytics を接続**（オーナー/cowork）。
6. **既存noteから内部リンク**で初速 → **X/Redditで控えめに告知**（オーナー/cowork）。
7. （順位が出てPVが育ったら）**AdSense/ASPに申請**して(a)(b)を後乗せ（オーナー/cowork）。
8. CAOが週1でSearch Console順位を確認 → 効くツールに集中、効かないものは作り直し/撤退。
