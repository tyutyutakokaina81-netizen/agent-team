## 2. 入金/税セットアップ手順

> ⚠️ 以下は**一般的な手順の説明**。各社の画面・項目名・しきい値・税制は変更される。**実際の登録時に各社の最新案内と公式情報を必ず確認**すること。金額・税務の最終判断は専門家（税理士）に相談。

### 2-1. Amazon KDP（最重要：米国30%源泉を回避する）

KDPは米国企業のため、設定を怠ると**支払いから一律30%が源泉徴収**される。日本在住者は日米租税条約により**この30%を軽減（ロイヤリティは通常0%相当まで）できる**。以下を出版前に完了する。

**(a) 銀行口座登録（国内銀行でOK）**
1. KDPアカウント → 「税・支払い情報」または「Payment（お支払い）」セクションを開く。
2. マーケットプレイスごと（Amazon.co.jp＝円、Amazon.com＝ドル等）に**振込先銀行口座**を登録。
   - 国内銀行口座で受取可能。登録項目の一般例：**銀行名・支店・口座番号・口座名義（半角ローマ字＝通帳の英字表記に一致させる）・銀行コード/SWIFT** が求められる場合がある。
   - 円のマーケット（co.jp）は円のまま国内口座へ振込まれる。**ドル等のマーケット（.com他）はKDPが円換算して国内口座へ振込む**（為替は各社レート＝§0-4）。
3. 名義は**通帳・キャッシュカードの表記と完全一致**させる（不一致だと振込が失敗する）。

**(b) 税務インタビュー＝W-8BEN（米国非居住者の申告）でマイナンバーを使い30%を回避**
1. KDPの「税に関するインタビュー（Tax Interview）」を開始。
2. 「米国人ですか？」→ **いいえ**（日本居住の個人）。種別は**個人（Individual）**を選択。
3. 様式は **W-8BEN**（個人の米国非居住者用）が自動的に選ばれる。
4. **租税条約の特典（Treaty Benefits）を「請求する（Claim）」を選ぶ**。
   - 居住国：**Japan**。日米租税条約に基づき、書籍ロイヤリティの源泉税率は**軽減（一般に0%相当）**になる。
5. **納税者番号（TIN）欄に「マイナンバー（個人番号）」を入力**する（Foreign TIN＝外国の納税者番号として日本のマイナンバーが使える）。
   - これにより米国TIN（ITIN/SSN）を取得しなくても条約特典が適用され、**30%源泉を回避**できる。
6. 電子署名（氏名・日付）して送信。完了後、**源泉税率が0%（または軽減後の率）に反映されているか**を税情報画面で確認。
   - ⚠️ もし税務インタビュー未完了／条約請求漏れだと**30%が引かれる**。出版前に必ず完了させる。
7. ⚠️ 様式名・条約の適用率・入力欄は変更されうる。**画面の最新案内に従う**。不明点は税理士に確認。

**(c) 最低支払いしきい値・支払いサイクル**
- KDPはマーケットごとに**最低支払額のしきい値**があり、達するまで繰り越される（通貨により金額が異なる）。**振込方式が銀行振込(EFT)なら、しきい値が低め**に設定される傾向。
- 支払いは概ね**販売月の約60日後**（例：1月販売分→3月末頃）に各マーケットから入金。
- ⚠️ しきい値・サイクルは変更あり。**出品時に最新を確認**。少額のうちは繰り越されて入金されない点に留意（赤字ではなく未達＝翌月以降に合算）。

### 2-2. Gumroad / PayPal の受取設定

**(a) Gumroad**
- Gumroadの売上受取は、地域により **(i)銀行振込（対応国）** または **(ii)PayPal経由** になる。日本は**PayPal受取**が基本ルート（直接の国内銀行振込が使えない場合がある）。
- 設定：Gumroad → Settings → Payments で**受取方法**を選ぶ。PayPalルートなら**PayPalアカウントのメールアドレス**を登録。
- Gumroadも**米国企業**＝Settings内の**税フォーム（W-8BEN相当）**の提出を求められる場合がある。求められたら §2-1(b)同様、**日本居住・条約特典・マイナンバーをForeign TINとして**入力。
- ⚠️ 受取可否・手数料・最低支払額は国とアカウント状態で変わる。**登録時にGumroadの最新案内を確認**。

**(b) PayPal（Gumroadの受取先 兼 単体決済）**
1. PayPal**ビジネス（またはパーソナル）アカウント**を作成（無料）。本人確認（氏名・住所・本人確認書類）を完了。
2. **国内銀行口座を紐付け**（PayPal残高→国内口座へ「引き出し」）。口座名義はPayPal登録名と一致させる。
3. **引き出し（出金）には最低金額・手数料の条件**がある場合がある。ドル残高を円で引き出す際は**PayPalの為替レート**が適用（§0-4＝参照¥150とずれる）。
- ⚠️ 引き出し手数料・為替手数料・最低出金額は変更あり。**実出金時に明細で確認**。

### 2-3. 為替・しきい値の共通注意

- ドル建て売上（KDP.com / Gumroad / PayPal）は**入金時に円換算**され、レート・タイミング・各社手数料で**手取りが参照計算（¥150）から数%〜10%程度ずれる**。確定額は**必ず入金明細で確認**。
- **最低支払いしきい値に達するまで入金されない**＝立ち上げ期は売上が「未入金で繰り越し」になりうる。これは赤字ではなく**入金タイミングの問題**。トラッカー（§4）では「発生ベース（売れた月）」で手取りを記録し、入金は別途確認する。

## 3. 確定申告メモ（断定せず・専門家確認前提）

> ⚠️ 以下は**一般的な考え方**であり、個別の最終判断ではない。所得区分・控除・申告要否は各人の状況で変わる。**必ず税理士・税務署・国税庁の最新情報で確認**すること。

**(a) 得た収益は申告対象になりうる【一般論】**
- これらの販売収益（Kindle印税・Gumroad/note売上等）で得た所得は、**確定申告の対象になりうる**。
- 一般に、給与所得者の副業的な所得が**年間20万円**を超える等の場合に申告が必要になる、といった基準が語られるが、**住民税の申告は別基準**であり、20万円以下でも申告が要るケースがある。**金額の線引きは断定せず、税理士・税務署に確認**する。

**(b) 所得区分（雑所得／事業所得）の一般的な考え方【一般論・断定しない】**
- 規模が小さく単発的なら**雑所得**、継続・反復・独立して事業的規模で営むなら**事業所得**と扱われる、という一般的な整理がある。
- どちらに当たるかは**継続性・営利性・規模**等で総合判断され、**自己判断で断定できない**。事業所得なら青色申告等の制度を使える余地があるが、**適用可否は専門家に確認**。

**(c) 経費になりうるものの一般例【一般論・要確認】**
- 売上を得るために直接要した費用は経費になりうる。**一般例**：
  - 本ラインの固定費に含むツール（Claude Pro／Canva／Microsoft 365）の**事業利用按分**。
  - 制作・販売に使う通信費・PC等の**事業按分**、出版・販売に関わる手数料。
- ⚠️ **家事按分（私用と兼用の費用の事業割合）**の考え方・割合は判断が要る。**領収書・明細を保存**し、計上可否と按分割合は**税理士に確認**。
- 経費区分の整理には自社テンプレ（`template_kit/csv/keihi_kubun_hayami.csv` 経費区分早見）が補助になる。

**(d) 外国源泉の扱い【一般論】**
- 米国マーケット分は §2-1(b) で源泉を回避（条約適用）するのが基本。万一源泉徴収された場合の**外国税額控除**の要否等も**税理士に確認**。

## 4. 売上トラッカー様式（商品別・月次）

> 目的：**商品別に 月次の 販売数／手取り／累計 を記録し、固定費¥5,800を超えたら黒字を自動判定**する。
> 記録は**発生ベース（売れた月）の手取り**で行う（入金は§2-3の通りタイミングがずれるため別途確認）。手取りは§0-3係数の【試算】、確定後は実額に置き換える。

### 4-1. 月次サマリ表（手取り合計が¥5,800超で黒字）

| 月(YYYY-MM) | 月間手取り合計（¥換算） | 固定費 | **月次損益（手取り−5,800）** | 累計損益 | 黒字判定 |
|---|---|---|---|---|---|
| 2026-07 | （記入） | 5,800 | =手取り−5,800 | =前月累計+当月損益 | 当月損益>0なら「月次黒字」 |
| 2026-08 | | 5,800 | | | |
| … | | | | | 累計損益>0なら「累計黒字」 |

- **月次黒字＝月間手取り合計 > ¥5,800**（その月の固定費を回収）。
- **累計黒字＝累計損益 > ¥0**（立ち上げ期の累積赤字を取り戻した）。
- ドル建て商品は**当月の入金明細レート（または参照¥150）で円換算**してから合算（換算根拠を備考に残す）。

### 4-2. 商品別 明細（CSVヘッダ例）

商品×月の明細を1行ずつ記録する。下記をそのまま `sales_tracker.csv` のヘッダに使える。

```csv
month,product_id,product_name,channel,currency,unit_price,units_sold,net_coeff,gross_net_local,fx_rate,net_jpy,cumulative_net_jpy,note
```

- `month`：YYYY-MM（売れた月）
- `product_id`：K1〜K6 / G1 / G2 / D1（プロンプト集）/ D2（テンプレ集）/ B1（バンドル）
- `channel`：KDP_JP / KDP_US / gumroad / note / booth
- `currency`：JPY / USD
- `unit_price`：販売単価（その通貨）
- `units_sold`：販売数
- `net_coeff`：手取り係数（KDP=0.70／Gumroad=0.88／note=0.72／BOOTH=0.85。⚠️最新確認）
- `gross_net_local`：手取り（その通貨）＝ unit_price × units_sold × net_coeff
- `fx_rate`：USD→JPY換算レート（JPYなら1。USDは入金明細のレート、暫定は150）
- `net_jpy`：円換算手取り＝ gross_net_local × fx_rate
- `cumulative_net_jpy`：商品横断の累計手取り（行の積み上げ）
- `note`：備考（為替根拠・導入価格適用・未入金繰越 等）

### 4-3. 黒字自動判定の考え方（表計算で組む）

- 月ごとに `net_jpy` を合計 → **`SUMIF(month列, 当月, net_jpy列)`** で月間手取り合計。
- **月次黒字判定**：`= IF( 月間手取り合計 > 5800, "黒字", "未達" )`
- **累計黒字判定**：当月までの月間損益（手取り合計−5,800）を累計 → `= IF( 累計損益 > 0, "累計黒字", "回収中" )`
- **赤字化の唯一の経路は固定費を¥5,800超に増やすこと**。新規課金（画像生成AI・有料広告・有料SaaS等）を入れない限り、固定費は¥5,800一定＝この判定式の基準は動かない。新規課金を検討するときは「黒字に必要な部数が増える」と必ず換算してから判断する。
- 既存テンプレ `template_kit/csv/gesshu_shushi.csv`（月収支）/ `cash_mitooshi.csv`（キャッシュ見通し）を流用すると、同一様式で家計・他収益と合算できる。
