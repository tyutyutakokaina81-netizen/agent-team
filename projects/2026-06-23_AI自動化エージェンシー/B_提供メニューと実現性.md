# B_提供メニューと実現性 — 「AI自動化代行」で実際に何を納品できるか

> 作成：CDO 兼 CPO（2026-06-23）
> 目的：「AI自動化代行」を看板にしたとき、**当社の物理制約で本当に納品できるもの**だけを、誇張せず具体化する。
> 姉妹資料：`projects/2026-06-23_300K設計/A_高単価B2Bパッケージ.md`（=英語発信に特化した縦の商品）。
> 本ファイルは**それより横に広い「AI自動化全般」の現実的メニュー表＋実現性の線引き＋標準パッケージ化**を担う。
> 制作：AI（code）が成果物を作る。外部操作（営業・契約・客先での常時稼働）はオーナー/cowork/客先。**ゼロ予算**。

---

## 0. 前提となる物理制約（毎回ここから設計する）

| # | 制約 | 設計上の含意（=売り方が決まる） |
|---|---|---|
| C-net | code（私）は**外部ネット不可**（A1：全403） | 客のサーバ・SaaS・APIに**当社が常時つなぐ運用は不可**。**作って渡す**形に限定。 |
| C-key | **APIキー課金の肩代わり不可**（A2隣接・ゼロ予算） | 客のAI利用料は**客が自分の口座/キーで払う**。当社は「設定・台本・テンプレ」を渡すだけ。 |
| C-srv | **客向けの常時稼働サーバ/SaaSを当社が運用するのは不可** | 監視・24h稼働・障害対応は範囲外。**客側で動く仕組み**を納品し、運用は客 or 客のツール側に委ねる。 |
| C-img | **画像生成不可**（A2） | サムネ・図版・写真は作れない。**文章・スクリプト・手順・テンプレ**が主成果物。図は客側ツール任せ。 |
| C-pii | **PII禁止**（A4） | 客の顧客名簿・実住所・個人名を当社ファイルに残さない。穴埋め欄〔　〕で渡し、客が手元で記入。 |
| C-cup | **コンテナ使い捨て**（A7） | 当社が状態を持つ運用は不可。**成果物＝静的ファイル**として一度に渡し切る。 |

> **結論（売り方の芯）**：当社の「AI自動化代行」は**運用代行(BPO)ではなく、"客が自社で回せる自動化の仕組み一式を設計・制作して納品する"代行**。
> = **「DFY（Done-For-You）の仕組み納品」＋「使い方トレーニング」**。常時稼働・監視・従量課金の肩代わりは**売らない**（できないものを売って炎上させない＝A5）。

## 1. 提供できる自動化メニュー（現実的・6種）

各メニュー：**何を渡すか／客側で何が要るか／できる・できないの線引き**を正直に。

### (a) AIプロンプト体系 ＋ GPT設定（客が自社で使う）

- **渡すもの**：業務別プロンプト集（コピペで使える日本語/英語）／カスタムGPT・Projects・システムプロンプトの設定文／使い分けの早見表／入力テンプレ（穴埋め式）。
- **客側で要るもの**：客自身のChatGPT/Claude等のアカウント（**課金は客**）。ブラウザのみ。
- **できる**：客の頻出業務（メール・要約・企画・翻訳・分類）を**プロンプト1枚で再現可能**にする。属人化の解消。
- **できない**：当社が客のアカウントに常駐して自動実行すること（C-srv/C-net）。**「貼って使う」運用まで**。

### (b) 文章/記事/SNS量産システム（テンプレ＋プロンプト＋手順）

- **渡すもの**：記事・SNSの**型テンプレ**（構成・トンマナ・禁止語）／量産プロンプト（1テーマ→N本）／投稿カレンダーの雛形／品質チェックリスト。
- **客側で要るもの**：AIアカウント／投稿先アカウント。投稿は**客が実行**（合意時のみ cowork が代行可、当社codeは不可）。
- **できる**：客が**自分で月20〜30本を回せる仕組み**を渡す。当社の `quality_check.py` 相当の検品ルールも同梱。
- **できない**：当社サーバからの自動投稿・予約投稿の常時運用（C-srv）。**仕組みは渡すが、回すのは客**。

### (c) 問い合わせ・メール返信の下書き自動化レシピ

- **渡すもの**：問い合わせ分類プロンプト／返信下書きテンプレ（状況別10〜20型）／英語問い合わせの翻訳＋下書きレシピ／**Gmail/スプレッドシート連携の手順書**（客側のGAS雛形コードを含む）。
- **客側で要るもの**：客のメール環境（Gmail等）。GAS（Google Apps Script）は**客のアカウント内で動く**＝当社サーバ不要。
- **できる**：受信→分類→下書き生成までの**レシピとGASコードを渡す**。客が自分の環境に貼れば客側で動く。
- **できない**：当社が客の受信箱を常時監視して自動返信すること（C-net/C-srv）。**最終送信は必ず客が承認**。

### (d) データ整形 / 議事録整形のスクリプト・手順

- **渡すもの**：**依存ゼロのPython/GASスクリプト**（CSV整形・名寄せ・集計・転記）／議事録→要約→アクション抽出のプロンプト＋手順／実行手順書（コマンド or スプレッドシート操作）。
- **客側で要るもの**：Python実行環境 or Googleスプレッドシート。データは客が用意（PIIは当社に渡さない＝C-pii）。
- **できる**：**手元で完結する変換スクリプト**を渡す（当社の `tools/*.py` と同じ「依存ゼロ・ネット不要」思想）。サンプル入出力で動作確認済みのものを納品。
- **できない**：客の本番DB/基幹システムへの常時接続・同期（C-net/C-srv）。**バッチ的な"手元で回す"までが範囲**。

### (e) 社内マニュアル・FAQのAI化

- **渡すもの**：既存マニュアル→Q&A化プロンプト／社内FAQナレッジの構造化テンプレ／「社内GPTに食わせる元データ」の整形手順／FAQ運用ルール。
- **客側で要るもの**：客のAI環境（社内利用可のプラン）／元になるマニュアル類。
- **できる**：散らばった社内文書を**AIが答えられる形（Q&A・構造化テキスト）に整形**して渡す。更新手順も同梱。
- **できない**：当社が社内チャットボットを**ホスティング/運用**すること（C-srv）。**「客の社内AIに載せる素材＋手順」まで**。

### (f) スプレッドシート自動集計テンプレ

- **渡すもの**：関数/GAS入りの**集計テンプレ**（売上・在庫・KPI・シフト等）／入力規則・ダッシュボード雛形／更新手順／（必要なら）AIで自動コメント生成するプロンプト。
- **客側で要るもの**：Googleスプレッドシート or Excel。
- **できる**：**開いて入力すれば集計される**テンプレを渡す。GASは客のアカウント内で動く＝当社サーバ不要。
- **できない**：当社サーバでの自動更新・外部API連携の常時運用（C-net/C-srv）。**テンプレ＋客環境で動くGASまで**。

## 2. できないこと（誠実な線引き・先に言う）

> 売り込み前に**このリストを客へ提示**する。後出しの「できませんでした」を構造的に防ぐ（A5＝誇張・できない約束をしない）。

| 範囲外 | 理由（制約） | 代替として当社が出せるもの |
|---|---|---|
| 24h常時稼働の監視・障害対応 | C-srv：当社は客向けサーバを運用しない | 客側で動く仕組み＋「異常時の対処手順書」 |
| 客のサーバ/インフラの構築・保守・運用 | C-net/C-srv：当社は客環境に常時接続できない | 構成案・設定手順・スクリプトの**納品**（実装は客 or 客のベンダー） |
| API従量課金の肩代わり・当社キーでの提供 | C-key/ゼロ予算 | 客自身のキー設定手順＋コスト試算（客が自分で契約） |
| 当社からの自動投稿・自動送信の常時運用 | C-net/C-srv | 投稿/送信の**下書き生成＋手順**（実行と最終承認は客 or cowork） |
| 画像・図版・動画素材の生成 | C-img/A2 | 文章・台本・構成。画像は客側ツール（Canva/画像AI）の**プロンプト＋手順**を提供 |
| 客の個人情報・名簿の預かり処理 | C-pii/A4 | PIIを当社に渡さず**客の手元で動く**スクリプト＋穴埋め欄〔　〕方式 |
| 成果（売上増/問い合わせ増/順位上昇）の保証 | A5 | 約束するのは**「仕組み・テンプレ・手順を計画的に作って渡し、使えるようにすること」**だけ |

## 3. 納品物の形（フォーマット）

| 形式 | 中身 | 制作 | 備考 |
|---|---|---|---|
| **ドキュメント**（.md/.pdf/.docx） | プロンプト集・手順書・FAQ・運用ルール・マニュアル | code | docx化は `tools/make_epub.py` 系・客側変換でも可 |
| **スクリプト**（.py/.gs/.js） | 整形・集計・連携の**依存ゼロコード**（GASは客アカウントで動く） | code | サンプル入出力付き・ネット不要で動作確認 |
| **テンプレ**（スプレッドシート/構成型/穴埋め） | 集計テンプレ・記事型・投稿カレンダー・入力フォーム | code | PIIは〔　〕で空け、客が手元で記入（C-pii） |
| **動画手順**（操作録画） | 「貼って使う」「実行する」の操作デモ | **オーナー**（codeは画面録画不可） | codeが**台本**を書き、オーナーが録画 |
| **トレーニング**（ライブ/Q&A） | 使い方レクチャー・初回伴走・質疑 | オーナー/cowork | codeが**進行台本＋想定Q&A**を用意 |

> 原則：**code = 静的成果物（文章・コード・テンプレ・台本）を全部作る**／**オーナー・cowork = 録画・対面・客先実行**。境界は§0の制約そのまま。

## 4. 標準パッケージ化（売れ筋3つを「型」にする）

> クライアント別の作り込みを最小化するため、**3つの型**に集約する。各型は **clients/<id>/ 構造 ＋ retainers.py ＋ quality_check.py** にそのまま乗せる（既存インフラ流用＝再現性）。
> 既存 `retainers.py` の plan（bronze/silver/gold）は英語発信代行の月次ノルマに最適化済み。**自動化代行は「単発の仕組み納品」が主**なので、下記は"単発パッケージ＋任意の保守"として定義し、月次が要るものだけ retainers の月次サイクルに載せる。

### 型① 「業務プロンプト＆GPT設定 一式」パッケージ（=メニュー a）

- **対象**：AIを使いたいが何をどう書けばいいか分からない中小・個人事業。
- **標準納品物（固定セット）**：①業務別プロンプト集15〜30本 ②カスタムGPT/Projects設定文 ③使い分け早見表 ④入力穴埋めテンプレ ⑤操作台本（録画用）。
- **作り込みを最小化する仕組み**：`clients/_template/` を複製→`brief/`に客の業務ヒアリングを記入→**プロンプト集の8割は共通骨子、2割だけ客業務に差し替え**。
- **保守**：原則**単発**（必要なら四半期アップデートを軽い保守として）。
- **検品**：`tools/quality_check.py` で禁止語・混入物（社内メモ/コードブロック痕）を検査してから納品。

### 型② 「コンテンツ量産システム」パッケージ（=メニュー b、月次保守と相性良）

- **対象**：自社で記事/SNSを継続発信したい事業者。
- **標準納品物**：①型テンプレ（記事・SNS）②量産プロンプト ③投稿カレンダー雛形 ④品質チェックリスト（quality_check相当）⑤初月の見本コンテンツ。
- **作り込みを最小化**：テンプレ・プロンプトは共通、**トンマナだけ `clients/<id>/style/style.md` で差し替え**（既存clients構造のstyleをそのまま使う）。
- **保守（任意・月次）**：月次の見本追加＋チェックは **retainers.py の月次サイクル**に載せる（open→制作→check→delivered→invoiced→paid）。
- **検品**：`retainers.py check`（内部で quality_check を呼ぶ）で毎月ゲート。

### 型③ 「データ整形/集計 自動化」パッケージ（=メニュー d＋f）

- **対象**：手作業のExcel集計・転記・整形に時間を取られている事業者。
- **標準納品物**：①依存ゼロ整形/集計スクリプト（Python or GAS）②集計テンプレ＋ダッシュボード雛形 ③実行手順書 ④サンプル入出力 ⑤操作台本（録画用）。
- **作り込みを最小化**：**スクリプトの骨格を共通ライブラリ化**し、客のカラム定義だけ設定ファイルで差し替える（当社 `tools/*.py` と同じ"依存ゼロ・ネット不要"思想で量産）。
- **保守**：原則単発（客環境で完結＝C-srv回避）。仕様変更時のみスポット。
- **検品**：サンプルデータで**実行して動作確認**＋手順書のセルフレビュー。

> **3型に集約する理由**：(a)(d)(f)は単発で売り切れて運用負荷が無い＝**当社の制約と最も相性が良い**（客環境で完結）。(b)だけ月次保守で継続課金にできる＝**retainers.py の継続課金モデルに合流**。(c)(e)は型②③の応用（レシピ/素材整形）として個別見積で対応し、まずは3型でカタログを固定する。

## 5. 品質保証（検品運用）

| 工程 | ツール/手順 | 合否基準 |
|---|---|---|
| 制作後セルフ検査 | `python3 tools/quality_check.py <納品ファイル>` | A5-HARD=0（誇張・断定ゼロ）・MIX=0（社内メモ/コード痕の混入ゼロ）|
| 客向け原稿の厳格検査 | `quality_check.py --reader <ファイル>` | 読者/客に出してはいけない混入物もHARD扱いで0 |
| 月次保守の検品（型②） | `python3 tools/retainers.py check --id <id> --month <YYYY-MM>` | 当月 delivered/ を一括検査・ノルマ充足を確認 |
| スクリプト系（型③） | サンプル入出力で**実行**→期待値一致を確認 | 例外なく完走・出力が仕様通り |
| PIIゲート（全工程） | quality_check のPII検出＋目視、穴埋め欄〔　〕の徹底 | 実名・住所・個人名の混入ゼロ（A4） |
| 納品前最終 | チェックリスト（型ごと固定）＋オーナー目視 | 全項目✓で「delivered」へ |

> 検品は**納品の必須ゲート**。HARD違反が1件でもあれば差し戻し（quality_check の終了コード1で機械的に止める）。継続案件は retainers.py の状態遷移（review→delivered）にこのゲートを組み込む。

## 6. 1案件の標準フロー（再現性の担保）

```
1. ヒアリング   : clients/_template/ を複製 → brief/ に客業務を記入（PIIは〔　〕）
2. 型を選ぶ     : 型①/②/③ から選択（複合は見積で）
3. 制作         : code が標準納品物セットを作る（共通8割＋差し替え2割）
4. 検品         : quality_check（＋型③は実行確認）→ HARD=0 を確認
5. 納品         : ドキュメント/スクリプト/テンプレを渡す。台本→オーナーが録画
6. トレーニング : オーナー/cowork が使い方レクチャー（codeが台本・想定Q&A）
7. （型②のみ）   : retainers.py で月次保守サイクルに載せる
```

---

## 7. 結論 — 最も売りやすい自動化メニュー

- **「業務プロンプト＆GPT設定 一式」パッケージ（型①）**が最有力。
- 理由：**単発で売り切れ・客環境で完結**＝当社の常時稼働/ネット不可の制約に一切抵触せず、運用負荷ゼロ。
- かつ「AIを使いたいが書き方が分からない」需要は広く、**8割共通テンプレ＋2割差し替え**で原価が極小＝粗利が高く再現性も高い。
