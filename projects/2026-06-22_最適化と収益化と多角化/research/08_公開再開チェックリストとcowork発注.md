# 公開再開チェックリストと cowork 発注ドラフト

作成日: 2026-06-22
担当: CDO（技術・自動化責任者）
目的: 2026-06-12 インシデント以降停止中の note 日次公開を、**安全に**再開するための実務手順を確定する。

---

## 0. 背景（事実の確認）

【事実】2026-06-12 に note 側で重複・テスト・未来日付記事の過剰公開と台帳乖離が発生し、Cowork が **69本を非公開化・12本削除**して一次収束。
【事実】2026-06-13 の日次公開タスクは、STATE.md 冒頭の「棚卸し完了まで自動公開を再開しない」指示に従い **1本も公開せず中止**。
【事実】再開条件は STATE.md 決定ログ（2026-06-13）に明記された次の3点。

1. note 側の重複/テスト/未来日付記事の棚卸し（**オーナー操作**）
2. 公開スクリプトの冪等化・未来日付ガード点検（**Claude Code = CDO**）
3. 台帳と note 実態の再同期（**Claude Code が判断、Cowork が実データ取得を代行**）

【考察】3点は「外部実体（note）」「スクリプト（コード）」「台帳（リポジトリ）」の三者の整合を取り直す作業であり、担当が物理的に分かれる。誰が確認するかを明示しないと再発する。
【提案】以下を「担当者つきチェックリスト」に分解し、全項目クリアを **再開のゲート条件** とする。

### 絶対制約の再掲（本タスクで特に効くもの）

- **A1**: 外部ネット全面ブロック。CDO（私）は note を直接確認・投稿できない。note 実体に関わる確認は **オーナー or Cowork** が行う。
- **A4**: 具体的な地区名・町名・住所を一切書かない（本ファイル・発注 YAML・台帳すべて）。
- **A7**: コンテナは使い捨て。確認結果・棚卸し結果は **必ずリポジトリ内ファイル（ops/outbox 等）に書き戻して commit** しない限り消える。口頭・チャットだけの報告は記憶として残らない前提で設計する。

## 1. 再開条件3点 → 担当者つきチェックリスト

> 凡例: ☐=未／担当を [ ] 内に明記。Owner=オーナー手動、Cowork=クラウド実行（ブラウザ/認証）、CDO=Claude Code（コード・判断・台帳）。

### 条件(1) note 側の棚卸し（主担当: Owner、実データ取得補助: Cowork）

- ☐ [Owner/Cowork] note 管理画面で **公開済み記事の全件リスト** を取得する（タイトル・note ID・公開日時・公開/非公開ステータス）。
- ☐ [Cowork] 取得した一覧を **機械可読な形式**（CSV か Markdown 表）にして `ops/outbox/` に書き戻す（A7: 書き戻さないと消える）。
- ☐ [Owner/Cowork] **重複記事**（同一タイトル・同一本文の二重公開）を抽出し、残す1本以外を非公開化する。
- ☐ [Owner/Cowork] **テスト記事**（"test"・ダミー・動作確認用）が公開残存していないか確認し、あれば非公開化または削除する。
- ☐ [Owner/Cowork] **未来日付記事**（公開日時が実行日より先のもの）が無いか確認する。あれば非公開化する。
- ☐ [Owner] 棚卸し後の **最終的な「正しい公開済み一覧」を確定**し、確定版である旨を `ops/outbox/` のレポートに明記する。
- ☐ [CDO] レポートを受領し、件数（公開N本／非公開化M本／削除K本）を本タスクの台帳側に記録できる状態にする。

【考察】A1 により私（CDO）は note を直接見られない。この条件は **構造的にオーナー/Cowork しか実行できない**。私の役割は「何を確認すべきか」のリスト提供と、結果の受領・台帳反映に限られる。

### 条件(2) 公開スクリプトの冪等化・未来日付ガード点検（主担当: CDO）

- ☐ [CDO] `publish_to_note.py` / `note_publisher` 系で **同一記事の二重公開を防ぐ冪等キー** があるか点検する（公開済みマーカー or published/ への移動が公開成功の単一の真実になっているか）。
- ☐ [CDO] **公開前に「ファイル名日付 or 本文日付 > 実行日」を弾く未来日付ガード**が有効か確認する（インシデント時に未来日付が流出した経路を塞ぐ）。
- ☐ [CDO] **テスト記事フィルタ**（ファイル名・本文に test/ダミー指標があれば公開対象から除外）が効いているか確認する。
- ☐ [CDO] **1日最大5本のハードリミット**が実装されているか確認する（B3=5本厳守と整合）。
- ☐ [CDO] **並列実行防止**（同時に2バッチが走らないロック）が効いているか確認する。
- ☐ [CDO] `drafts/queue/ → drafts/published/` のステージング移動が **公開成功時のみ・原子的に** 行われるか確認する（途中失敗で queue に残る＝再実行で二重公開、を防ぐ）。
- ☐ [CDO] 点検で穴が見つかった場合は **コードを修正**し（実行ではなく静的点検＋修正）、修正点を本タスク内の別researchか PR で記録する。
- ☐ [CDO] スクリプトの **dry-run（公開せず対象一覧だけ出力するモード）** が使えるか確認する。無ければ追加を提案する（再開初回は dry-run で安全確認したい）。

【考察】この条件は私が単独で完結できる唯一の条件。ただし**実行検証（実際に publish を走らせる）は A1 によりできない**ため、点検は静的レビュー＋ロジック確認＋dry-run 設計までが上限。実走の確認は Cowork の dry-run 実行に委ねる。

### 条件(3) 台帳と note 実態の再同期（判断: CDO、実データ: Cowork/Owner）

- ☐ [Cowork/Owner] 条件(1)で確定した「note の正しい公開済み一覧」を `ops/outbox/` 経由で CDO に渡す（A7: ファイルで渡す）。
- ☐ [CDO] `CMO/_index.md`（台帳・正本）の公開ステータスと、note 実態一覧を **突き合わせる**。
- ☐ [CDO] 乖離を分類する: ①台帳「公開済」だが note に無い ②note にあるが台帳「未公開」 ③note ID 未記録 ④日付不一致。
- ☐ [CDO] 各乖離を **台帳側を正に修正**（note 実体を真実とし、台帳をそれに合わせる。台帳は実ファイル走査で再生成可能な正本）。
- ☐ [CDO] 突き合わせ後、**「次に公開してよい未公開ストックの一覧」** を確定する（既に note 上にあるものを再公開しないため）。
- ☐ [CDO] 再同期完了を STATE.md に記録できる状態にする（※本タスクでは STATE.md は編集しないが、再開実施フェーズでオーナー承認後に更新する前提）。

【提案】突き合わせのキーは **note ID を第一キー、タイトル＋公開日を第二キー** とする。タイトルだけだと重複検出に弱いため、棚卸しレポートに **note ID を必須項目**で含めるよう Cowork に依頼する（下記発注 YAML に明記）。

### ゲート判定

- ☐ 条件(1)(2)(3) の全チェックが ☐→☑ になったか。
- ☐ [Owner] 最終 go/no-go をオーナーが承認したか（公開は外向き・センシティブのため最終判断はオーナー）。
- ☐ 上記2つが揃って初めて「2. 再開後の安全運用」に進む。

## 3. 再開後の「1日5本公開」安全運用手順

【提案】棚卸し直後にいきなり定常5本へ戻すと再発リスクが高い。**段階的ランプアップ**で安全域を確認してから定常運用に入る。

### 3-1. ランプアップ（再発防止のための慣らし運転）

| 日 | 公開本数 | 目的 |
|---|---|---|
| Day 1 | dry-run → 1本 | スクリプト・ガード・台帳反映の単発確認（発注その3） |
| Day 2 | 2〜3本 | 連続公開でも重複/未来日付が出ないか確認 |
| Day 3 以降 | 5本（定常） | B3=5本厳守・B4=食2本の定常運用へ移行 |

【考察】Day 1 を1本に絞るのは、インシデントの主因（重複・未来日付・テスト流出）が **公開1回ごとのガード**で防げるかを最小コストで検証するため。1本で問題なければ多重化しても同じガードが効く。

### 3-2. 毎日の公開前チェック（5本定常時・恒久ルール）

公開バッチを Cowork に投げる前に、CDO（私）が staging 段階で必ず確認する:

- ☐ [CDO] 対象5本の **ファイル名日付＝公開対象日** で、未来日付が混ざっていない。
- ☐ [CDO] 5本のタイトル・本文が **互いに重複しない**／既 published と重複しない（台帳照合）。
- ☐ [CDO] **食2本**（B4）・**同一カテゴリ2日連続回避**（B5）を満たす。
- ☐ [CDO] テスト/ダミー記事が queue に紛れていない。
- ☐ [CDO] `drafts/queue/` に**ちょうど5本**だけ存在する（余剰＝二重公開リスク）。
- ☐ [CDO] PII（具体的な地区名・町名・住所）が本文・タイトルに無い（A4）。

### 3-3. 公開実行（Cowork、二段構え）

- 通常: Cowork が `drafts/queue/` を pickup → dry-run で対象確認 → 公開 → `drafts/published/` へ移動 → `ops/outbox/` に結果書き戻し。
- 不調時: オーナーが `publish_to_note.py` で手動公開（North Star=海外配信を1点に依存させない）。
- **1日5本のハードリミット**をスクリプト側でも担保（人為ミスでも6本目が出ない）。

### 3-4. 公開後チェック（毎日・台帳整合の維持）

- ☐ [CDO] `ops/outbox/` の公開結果（note ID・公開日時）を受領する。
- ☐ [CDO] `CMO/_index.md` を「公開済」＋ note ID で更新する（台帳=正本を実態に追従）。
- ☐ [CDO] 公開本数が **5本ちょうど**であることを確認（多→二重公開疑い／少→失敗）。
- ☐ [CDO] STATE.md に当日の公開実績（件数・note ID）を追記して **commit & push**（A7: push しないと記憶が消える）。

### 3-5. 異常時の即時停止（サーキットブレーカー）

【提案】次のいずれかを検知したら **即日公開を止め、棚卸し（条件1）からやり直す**:

- 同一タイトル/本文が note 上に2件以上現れた（重複再発）。
- 未来日付の記事が公開された。
- 1日の公開本数が6本以上になった。
- `drafts/published/` と `CMO/_index.md` の公開済み件数が乖離した。

【考察】2026-06-12 の規模（69本非公開化）まで悪化したのは、検知が遅れて一括蓄積したため。**毎日の公開後チェックを軽量なサーキットブレーカーにしておけば**、再発しても1日分(最大5本)で止められる。

【事実】本ファイルは **チェックリストと発注ドラフトの設計物**であり、実際の棚卸し・公開・台帳更新・STATE.md 更新・commit は含まない（本タスクの厳守事項に従い、_index.md・STATE.md・既存ファイルは未編集、git 操作なし、ops/inbox に実ファイルなし）。
