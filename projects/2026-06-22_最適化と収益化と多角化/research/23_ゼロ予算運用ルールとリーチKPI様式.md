## 0. なぜCFOがこれを持つか（役割の再定義）

【事実】
- 2026-06-22 オーナー決裁で会社方針が確定：本命＝発信、儲けは狙わない、**お金はかけない（ゼロ予算）**、赤字にしない＝続ける。
- 収益化・有料販売・広告営業の設計（research 01-13）は「凍結（来たら使う）」。動かすのは無料発信のみ。
- 固定費は¥5,800/月（CLAUDE.md・STATE.md記載のランニングコスト）。

【考察】
- 収益を追わない会社では、CFOの仕事は「稼ぐ計画」ではなく「**増やさない規律**」と「**続けられる証拠（数字）の管理**」に移る。
- 黒字/赤字の定義もシンプル化する：収益≒0前提なので、**支出が¥5,800/月を超えないこと＝健全**。1円でも新規課金が入れば即「赤字化リスク」。

【提案】
- CFOの守備範囲を2つに固定する：
  1. **コスト規律**＝新規課金の門番（§1）／お金をかける判断の基準（§2）。
  2. **無料リーチKPIの器**＝数字を入れる様式の整備と健全性チェック（§3-§4）。**「なぜ効いたか」の分析はやらない**（=CAO領域）。

### 0-1. CAOとの棲み分け（被り防止・明記）

| 観点 | CFO（本ファイル） | CAO（research/10 ほか） |
|---|---|---|
| 主担当 | コスト規律＋数字の**器**（様式・台帳のフォーマット） | 反響の**分析**（なぜ効いたか・次の打ち手） |
| 問い | 「お金をかけていないか／続けられるか」 | 「どの記事・施策がなぜ伸びたか」 |
| KPI | 固定費・新規課金・無料リーチの素の数字を**正しく記録** | その数字を**解釈し仮説化**してCMO等へ供給 |
| 成果物 | 運用ルール／承認基準／スコアボード様式（空欄ひな型） | 分析レポート／インサイト／A/Bテスト設計 |
| 数字への態度 | 「測れて・記録できる」こと（器の正確性） | 「測って・動ける」こと（打ち手への翻訳） |

> 一言：**CFOは「数字を入れる箱を作って番をする人」、CAOは「箱の中身を読んで次を決める人」**。
> research/10（CAOのKPIダッシュボード）は収益指標と打ち手対応表を含む上位互換。本ファイルは収益を外し、
> ゼロ予算順守と無料リーチの素データ様式に絞る（重複を避ける）。月次の素の数字はここで器を定義し、解釈はCAOへ渡す。

## 1. ゼロ予算運用ルール（新規課金は原則禁止）

### 1-1. 現状の固定費¥5,800/月の内訳点検

【事実】
- STATE.md／CLAUDE.md に「ランニングコスト¥5,800/月」と記載があるが、**内訳の明細は社内に未整備**（金額の正本はCFO管理だが、本メモには明細額を確定記載しない＝CFO規約・_index.mdにも金額を書かない）。

【様式】内訳点検テンプレ（金額はCFOの非公開台帳で管理。本メモは項目の棚卸し用に空欄ひな型のみ置く）

| # | 費目 | 用途 | 月額 | 区分 | 無料代替の有無 | 解約しても発信は続くか |
|---|---|---|---|---|---|---|
| 1 | （例：ドメイン） | （例：Web/フリーペーパー配布先） | [CFO台帳] | 固定 | 要調査（無料サブドメイン等） | 要判定 |
| 2 | （例：ホスティング/Pages） | サイト公開 | [CFO台帳] | 固定 | GitHub Pages＝無料 | はい |
| 3 | （例：その他SaaS） | — | [CFO台帳] | 固定 | 要調査 | 要判定 |
| … | … | … | … | … | … | … |
| 合計 | | | **¥5,800（既知の総額）** | | | |

> 運用：オーナー/coworkがこの表を埋める（実際の月額はCFO非公開台帳に転記）。
> 目的は「**¥5,800の中身を全部言える状態**」にすること。説明できない費目＝即・廃止候補。

【提案・点検の問い（各費目に当てる3問）】
1. これを止めたら**発信（North Star）は止まるか**？ → 止まらないなら廃止候補。
2. **無料代替**はあるか？（GitHub Pages・無料サブドメイン・無料SaaS枠）→ あれば移行検討。
3. 年払い/自動更新になっていないか？ → 不要更新の事故を防ぐため更新日を§4の台帳に記録。

### 1-2. 「これ以上お金を増やさない」運用ルール（憲法）

【提案】以下を社内ルールとして固定する。

- **R-1（新規課金 原則禁止）**：いかなる新規の有料サブスク・買い切り・従量課金も**原則禁止**。既定の答えは「No / まず無料代替を探す」。
- **R-2（無料代替を必ず先に探す）**：有料を検討する前に、**無料手段の調査を1回必ず挟む**（GitHub Pages・無料ホスティング枠・OSS・既存無料ツール）。「無料で無理」を文書で示せて初めて有料を議題にできる。
- **R-3（上限固定）**：固定費の上限は**¥5,800/月を超えない**。新規費目を足すなら、同額以上の既存費目を削って総額を据え置く（**増額ゼロ＝ネット増を作らない**）。
- **R-4（無料枠は無料のまま）**：無料プラン利用中のツールが有料転換を促してきても、**自動でアップグレードしない**。無料枠内で使い切る運用を優先。
- **R-5（自動更新の監視）**：年払い・自動更新の更新日を台帳（§4）に記録し、**更新の30日前に要否を再判定**。惰性更新を禁止。
- **R-6（収益が来ても増やさない）**：方針どおり収益は「来たら拾う」。臨時収益が出ても、それを理由に固定費を増やさない（収益は黒字バッファに置く。支出膨張のトリガにしない）。
- **R-7（記録必須）**：課金・解約・無料代替への移行はすべて§4台帳に1行記録する（A7：記録しなければ消える）。

## 2. 万一お金をかける判断が要る時の基準

【事実】
- 方針上、新規支出は原則禁止（§1）。ただし「ドメイン更新が切れるとサイトが落ちる」等、**発信を止めないための最小支出**が要る場面は起こりうる。

【提案】例外を認める時の基準（ゲート方式）。下記を**全て満たす時だけ**議題化できる。

### 2-1. 支出ゲート（4条件 すべて必須）

1. **発信が止まる/質が著しく落ちる**ことの回避に直結する（あれば便利、ではダメ）。
2. **無料代替の調査を済ませ**、無理であることを一言で説明できる（R-2の証跡）。
3. **総額¥5,800/月を超えない**（超えるなら既存費目を削って相殺＝R-3）。
4. **オーナーが承認**（下記の承認テーブル）。

### 2-2. 承認テーブル（いくらまで／誰が）

| 区分 | 金額（試算・目安） | 承認者 | 備考 |
|---|---|---|---|
| ¥0（無料代替で完結） | ¥0 | 承認不要（CFOが記録のみ） | 既定の正解。R-7で台帳記録だけする |
| 既存費目の範囲内の付け替え | 総額据え置き（ネット増¥0） | **CFO判断可**（オーナーへ事後報告） | 例：費目Aを解約し同額の費目Bへ。総額不変が条件 |
| 総額が増える支出（恒常） | 月額ベースで**ネット増が出る**もの | **オーナー承認必須** | 金額の大小に関わらず「総額¥5,800を超える方向」は必ずオーナー |
| 一時的な少額支出（単発・非恒常） | **目安 ¥1,000/件・¥2,000/月 まで** | **オーナー承認必須**（CFOが起案） | ドメイン更新等。月内合算が目安超なら再協議 |

> 【試算】上記の¥1,000/¥2,000 は「ゼロ予算の精神を壊さない少額」の目安であり確定値ではない。
> オーナーが基準額を変えたい場合は決定ログに残す。**判断に迷ったら必ずオーナーへ＝増やさない方向**。

### 2-3. 起案フォーマット（CFO→オーナー、1案件1ブロック）

```
【支出起案】
- 目的（何が止まる/落ちるのを防ぐか）：
- 無料代替の調査結果（なぜ無料で無理か）：
- 金額：月額/単発の別、 [CFO台帳に実額]、区分（§2-2のどれか）
- 総額への影響：¥5,800に対し +/- いくら、相殺の有無：
- CFO判定：承認推奨 / 保留 / 却下推奨
- オーナー決裁：□承認 □却下 □保留（理由：       ）
```

## 3. 無料リーチKPI 簡易スコアボード様式（月次）

【事実】
- A1により、note・X・Reddit・Web等の数字は**私（CFO/コンテナ）からは自動取得できない**。
- 取得できるのは**オーナー or cowork が各管理画面を見て手入力した数字だけ**（A7：コミットしないと消える）。

【考察】
- ゆえにスコアボードは「自動集計」ではなく「**手入力台帳＋空欄ひな型**」が正解。
- 入力が重いと続かない。**月1回・5〜10分で埋まる粒度**に絞る。**取れた数字だけ埋め、空欄OK**。空欄理由を一言だけ書く。
- ここで持つのは「**無料リーチの素の数字**」のみ。収益はCAO/CFO本台帳側（本メモには載せない＝§0-1棲み分け）。

【様式】月次スコアボード（`ops/logs/reach/YYYY-MM_reach.md` に1ファイル/月で運用する想定。以下は空欄ひな型）

### 3-1. ヘッダ

```
# YYYY-MM 無料リーチ・スコアボード
入力者：オーナー / cowork
入力日：YYYY-MM-DD（締めは翌月5日目安）
集計対象期間：YYYY-MM-01 〜 YYYY-MM-末
取得元メモ：note アナリティクス画面 ほか（A1：CFOは自動取得不可、手入力）
```

### 3-2. コア表（毎月必ず埋める・L1）

| # | 指標 | 定義 | 取得元（手入力） | 今月値 | 前月値 | メモ/空欄理由 |
|---|---|---|---|---|---|---|
| K1 | 海外読者数（月） | 海外からの閲覧ユーザー数（推定可・概数可） | note等の地域別アナリティクス | | | |


## 4. コスト規律 台帳（イベント記録・追記式）

【様式】固定費の変更・課金・解約・無料移行・更新日を1行ずつ記録（`CFO`非公開台帳に実額、本ひな型は項目定義）。

| 日付 | 種別（課金/解約/無料移行/更新/起案） | 費目 | ネット増減（+/-/±0） | 承認者 | 総額への影響（¥5,800±） | メモ |
|---|---|---|---|---|---|---|
| | | | | | | |

> 運用：§1のR-7と§2の起案はすべてここに着地する。**更新日（自動更新の30日前アラート用）も同じ台帳に書く**。
> 実額はCFO非公開台帳・本表は「何が起きたか」の履歴（金額は[CFO台帳]参照とし、_index.md/本メモに金額を書かない＝CFO規約）。

## 5. 運用フロー（誰が・いつ・どこへ）

【提案】

1. **月次（翌月5日目安）**：オーナー/cowork が §3 スコアボードを `ops/logs/reach/YYYY-MM_reach.md` に手入力 → commit。
2. **CFO（器の番）**：入力値の整合チェック（%が0-100か・桁・前月比の符号）。**解釈はしない**。伸びた上位記事を§3-5でCAOへ引き継ぐ。
3. **随時（支出が絡む時）**：§2のゲート→起案→オーナー決裁→§4台帳に記録。
4. **更新日の30日前**：§4台帳の更新日を見て、CFOが要否をオーナーへ確認（惰性更新禁止＝R-5）。
5. すべて**コミット必須**（A7）。空欄・欠測は責めない。続けることを最優先にする。

## 6. 3行要約

- CFOの新任務は「ゼロ予算を守る番人（新規課金は原則禁止・無料代替を先に探す・総額¥5,800を超えない）」＋「無料リーチの数字の器（月次スコアボード様式の整備）」で、**収益は追わず「来たら拾う」方針に整合**させた。
- お金をかける判断は4条件ゲート（発信維持に直結／無料代替が無理／総額据え置き／オーナー承認）で絞り、**総額が増える方向は金額の大小によらず必ずオーナー承認**とした。
- 無料リーチKPIは A1ゆえ手入力前提の空欄ひな型（海外読者数・海外比率・英語経由・上位国＝A4で国/地域レベルまで・公開本数・上位記事）として用意し、**CFO＝数字の器、CAO＝なぜ効いたかの分析**と棲み分けを明記した。
