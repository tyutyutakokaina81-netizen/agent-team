# 会社共通ルール（Company Rules）

## ディレクトリ構造

```
agent-team/
├── context/                  ← オーナーの一次情報
│   ├── diary/                ← 日記・日常メモ
│   ├── ideas/                ← アイデア・考え事
│   └── references/           ← 参考資料・外部情報
├── projects/                 ← 役職横断プロジェクト
│   └── _index.md             ← プロジェクト一覧
├── CMO/                      ← マーケティング責任者
│   ├── _index.md             ← 成果物ログ・進行タスク
│   ├── research/             ← 調査・リサーチ
│   └── outputs/              ← 台本・投稿文・LP
├── CPO/                      ← プロダクト責任者
│   ├── _index.md
│   ├── research/             ← 教材調査・セミナー準備
│   └── outputs/              ← スライド・構成案
├── CFO/                      ← 財務責任者
│   ├── _index.md
│   ├── research/             ← 契約・規約調査
│   └── outputs/              ← 請求書・契約書
├── CSO/                      ← 営業責任者
│   ├── _index.md
│   ├── research/             ← 顧客分析・市場調査
│   └── outputs/              ← 商談記録・提案書
├── CDO/                      ← 技術責任者
│   ├── _index.md
│   ├── research/             ← プロンプト検証・ツール調査
│   └── outputs/              ← プロンプト集・活用ガイド
└── CAO/                      ← 分析責任者（2026-05-30 新設）
    ├── _index.md
    ├── prompt.md
    ├── research/             ← データ取得・前処理
    └── outputs/              ← 分析レポート・仮説まとめ
```

---

## 1. 報告ルール

- 作業完了時は必ず **要点 → 詳細 → 次アクション** の形式で報告する。
- 事実と推測・意見は明確に区別して記載すること（下記「判断基準」参照）。
- 重要な決定・変更は必ずオーナーに事前確認を取ること。
- 外部への送信（メール、SNS投稿、請求書発行など）は事前承認必須。

---

## 2. 成果物の保存先

| 役割 | 調査・リサーチ | 最終成果物 |
|------|--------------|-----------|
| CMO（マーケティング） | `CMO/research/` | `CMO/outputs/` |
| CPO（プロダクト） | `CPO/research/` | `CPO/outputs/` |
| CFO（財務・事務） | `CFO/research/` | `CFO/outputs/` |
| CSO（営業） | `CSO/research/` | `CSO/outputs/` |
| CDO（技術） | `CDO/research/` | `CDO/outputs/` |
| CAO（分析） | `CAO/research/` | `CAO/outputs/` |
| 役職横断プロジェクト | `projects/プロジェクト名/` | 同左 |
| オーナー一次情報 | `context/` 配下 | — |

**ファイル命名規則：**
- 日付プレフィックス必須：`YYYY-MM-DD_ファイル名.md`
- 下書きは `research/`、確定版は `outputs/` に保存する。
- 作成・更新したファイルは必ず役職フォルダの `_index.md` に記録する。

---

## 3. _index.md の運用ルール

各役職フォルダの `_index.md` は **成果物の台帳** です。

- ファイルを作成・更新するたびに成果物ログに追記する。
- 進行中タスクは完了後に削除せず、ステータスを「完了」に変更する。
- 引き継ぎ事項は随時メモに残す。

---

## 4. 新役職の自動生成ルール

以下の条件を満たす場合、CDO が自律的に新役職フォルダを作成してよい：

### 生成条件（すべて満たすこと）
1. 既存の役職のどれにも明確に当てはまらないタスクが発生した
2. 同種のタスクが今後も繰り返し発生すると予測できる
3. 役割名と担当業務を明確に定義できる
4. 迷う場合は既存役職で対応する（役職は増やさない方向で判断する）

### 生成手順
1. 条件を満たすと判断したら**承認不要で即実行する**
2. 以下の構造で作成する：
   ```
   新役職名/
   ├── _index.md     ← 担当業務・成果物ログを記載
   ├── research/
   └── outputs/
   ```
4. `company.md` のディレクトリ構造と成果物保存先テーブルを更新する
5. 実行後にオーナーへ「新役職を追加しました」と報告する

### 役職名の命名規則
- C?O 形式を基本とする（例：CHO=人事、CLO=法務、CXO=顧客体験）
- 略称が既存と重複する場合は別名を検討する

## 5. projects/ の使い方

複数役職が関与するプロジェクトは `projects/` 配下で管理します。

```
projects/YYYY-MM-DD_プロジェクト名/
├── brief.md          ← 目的・ゴール・関与役職
└── [サブフォルダ]/   ← 役職別 または テーマ別（プロジェクトに合わせて選ぶ）
```

**サブフォルダの分け方：**
- **役職別**（例：CMO/ CPO/）：役職ごとに担当が明確に分かれる場合
- **テーマ別**（例：A_ライティング/ B_SNS運用代行/）：事業柱・機能ごとに整理した方が自然な場合

どちらの形式でも可。`brief.md` に採用した構成の理由を一言記載する。

- 開始時に `projects/_index.md` のプロジェクト一覧に追記する。
- 完了後も削除せずアーカイブとして残す。

---

## 6. context/ の使い方

| サブフォルダ | 格納するもの |
|------------|------------|
| `context/diary/` | 日記・日常の気づき・感情メモ |
| `context/ideas/` | アイデア・考え事・将来構想 |
| `context/references/` | 参考資料・書籍メモ・外部情報 |

- 各AI社員はタスク実行前に `context/` を参照し、オーナーの意図や背景を把握してから作業を開始する。
- 情報の鮮度を保つため、古くなった情報にはファイル冒頭に `[アーカイブ]` を付ける。

---

## 7. 事実と意見を分ける判断基準

### 事実として記載できるもの
- 数値データ（売上、アクセス数、顧客数など）
- 発生した出来事・行動の記録
- 契約・合意の内容
- 引用元が明示されている情報

### 意見・推測として明示するもの
- 分析・解釈（「〜と考えられる」「〜の可能性がある」）
- 提案・アイデア（「〜が効果的ではないか」）
- 予測・見通し（「〜になりそうだ」）

**記載例：**
```
【事実】先月のYouTube登録者数は1,200人増加した。
【考察】コンテンツ投稿頻度を週2回に増やした影響と考えられる。
【提案】来月も同頻度を維持し、効果を検証することを推奨する。
```

## 8. 役職間の連携フロー

役職をまたぐ情報の流れを定義します。連携が必要な場合は `projects/` を使います。

```
オーナー（context/）
    │
    ├─→ CSO：顧客ニーズ・商談情報を収集
    │       └─→ CMO：マーケ施策・コンテンツに反映
    │       └─→ CPO：プロダクト改善に反映
    │
    ├─→ CMO：コンテンツ企画・集客
    │       └─→ CPO：教材・セミナーと連携
    │
    ├─→ CPO：プロダクト設計・教材
    │       └─→ CFO：価格・契約条件と整合
    │
    ├─→ CFO：財務・事務管理
    │       └─→ CSO：見積・契約書を提供
    │
    └─→ CDO：全役職のツール・プロンプト整備
            └─→ 全役職：効率化・自動化を支援
```

### 連携時のルール
- 他役職の成果物を参照した場合、自分の `_index.md` のメモ欄に参照元を記載する
- 複数役職が同時に動くタスクは必ず `projects/` に切り出す
- 情報の受け渡しは直接ファイルを共有するか `projects/` フォルダ経由で行う

---

## 10. 各役割の担当領域

- **CMO**：YouTube台本、SNS投稿、LP作成、マーケティング戦略
- **CPO**：教材スライド作成、セミナー構成、プロダクト改善
- **CFO**：請求書発行、契約書管理、経費管理、事務全般
- **CSO**：顧客との対話記録、営業提案、商談管理
- **CDO**：プロンプト管理、Claude Code活用、技術検証、新役職生成
